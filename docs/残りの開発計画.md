## 3. A: Windows 実機検証 —【最優先・中間ゴール成立の唯一の関門】

コード実装は揃っているが、開発環境が Linux コンテナのため **UIA・Electron・canvas・
グローバル入力フックに依存する部分は実機で目視・動作確認できていない**。ユーザー方針で
「全開発完了後にまとめて実施」とされており、いま実装が一段落した以上、**ここが唯一かつ
最大のボトルネック**。

### 検証チェックリスト（実機で確認する項目）

- [ ] **2-R2 UIA＋文生成**: クリック→要素解決→生成文が妥当か。Chrome ページ内解決の
      取得率を再計測（2-R0 では 75%）
- [ ] **2-R2b 操作拡張**: ダブルクリック・メモ帳/Excel への入力・Ctrl+S/F5・
      ドラッグON での D&D・Excel→Word 切替の appChange
- [ ] **2-R3 自動ズーム・枠**: 枠ハイライトと拡大画像の品質（文字が読めるか）
- [ ] **2-R4 取り込みウィザード**: 録画 → ウィザード → 取り込みの通し
- [ ] **3-R6 ガイドオーバーレイ**: 小窓の表示・ドラッグ移動・折りたたみ・撮影除外・
      録画ガジェットとの同時利用
- [ ] **3-A Excel/CSV**: 保存ダイアログ・Excel 本体での見た目・画像配置・CSV 文字化けなし
- [ ] **3-B 画像画質**: 原寸で文字が潰れないか・印刷/Word/PDF/Excel の見た目・容量増の実測
- [ ] **3-D PDF直接出力**: 保存ダイアログ・A4・背景印刷

### 中間ゴールの受け入れ基準（測定して初めて「成立」と言える）

ロードマップ「到達レベルの定義」より。**主観にしないための数値指標**:

- [ ] **手直し時間**: 録画停止 → 取り込み完了までの編集が、10手順程度で **5分以内**
- [ ] **自動生成文の採用率**: 無修正で使える生成文が **7割以上**（フォールバック文含む）
- [ ] **画像品質**: 全ステップで「全景＋クリック箇所の拡大」の2枚が人手ゼロで揃う

> これらが確認できた時点で **B3（中間ゴール成立の宣言）** が可能になる。

## 4. B: ユーザー判断待ちの意思決定

Claude が独断で決めない領域（運用ルール 領域3・4）。**次アクションはユーザーの判断**。

### B1. 3-C docx 移行の実装（`spec-3-C-docx-migration.md`）
- 調査結論: `docx` パッケージへ移行すれば後処理ハック `docx-postprocess.js`（242行）を
  丸ごと撤廃でき、Word「修復要求」の温床も構造的に消える。**総合的に価値は高い**。
- 主コスト: リッチ本文 HTML（`it.body`）の docx モデル変換と画像寸法計算。
- 回帰基準として `test/docx.test.js`（OOXML 骨格の固定）を新設済み。移行するならこれを
  満たしながら段階移行する。
- **判断事項**: 実装するか / 時期はいつか。

### B2. 3.5-A コード署名の購入（`spec-3.5-A-code-signing.md`）
- **決定済み（2026-07-14）**: 費用ゼロ方針のため購入見送り。無署名配布を継続し、
  無償の緩和策（SmartScreen 手順明記・SHA-256 チェックサム添付・ソース公開）のみ実施。
- **再検討トリガー**: Azure Artifact Signing（旧 Trusted Signing）の日本開放時。
- 注意: 証明書の CN を変えると自動更新（3.5-B）が壊れる。**CN 固定 → 3.5-B の順**を守る。

### B3. 中間ゴール成立の宣言

## 5. C: 未着手の実装項目（推奨順）

依存の薄いもの・効果の大きいものから。**S 規模はフェーズ2〜4の合間に差し込み可**。

### すぐ着手できる小粒（独立・低リスク）
- **C1 / 2-R2b 残 — ウィザードのセクション自動分割（S）**: 前面アプリ切替（appChange、
  サイドカー v4 に記録済み）を使い、取り込みウィザードで「ここから Excel の作業」と
  章立てを自動生成する。データは既に取れているので**分割ロジックのみ**。
- **C6 / 5-1 全文検索（S）**: ホームに検索ボックス。タイトル・項目・メモを横断する
  インクリメンタル検索。全件メモリ上のため**索引不要の線形走査**で十分。ヒットの
  ハイライトとエディタ内ジャンプまでやると完成度が高い。
- **C8 / 5-3 キーボード・a11y（S〜M）**: `Ctrl+F`（検索）・`Enter`（項目追加）・
  `Ctrl+P`（印刷）・`Esc`（閉じる、一部実装済み）。モーダルのフォーカストラップ、
  aria ラベル、両テーマのコントラスト検証（WCAG AA 目安）。

### 中粒
- **C7 / 5-2 タグ＋アーカイブ（M）**: model に `tags: string[]` と `archived: bool` を
  追加（旧データは `Store` 初期化時のデフォルト補完＝既存パターン）。フォルダ階層より
  先にタグで様子を見る。
- **C3 / 3.5-C リリース工程（S）**: フェーズ完了とバージョン番号の対応付け・CHANGELOG 整備・
  GitHub Releases にビルド済みインストーラを添付する運用（`build-windows.yml` が土台）。

### 大粒・要判断／前提あり
- **C2 / 3.5-B 自動更新（M）**: electron-updater＋GitHub Releases。起動時チェック→
  ユーザー確認の上で適用（サイレント強制更新はしない）。**実運用はコード署名（B2）が
  前提**の構成が望ましいため、署名の方針が決まってから着手するのが順当。
- **C4 / 4-R7 ローカルLLM文章仕上げ（L）**: node-llama-cpp＋小型日本語モデルを
  **任意ダウンロードのオプション**（既定無効）として統合。用途は文の自然文化・
  タイトル/概要の下書き・一括リライトに限定。推論も完全ローカル。核（R2）は LLM 無しで
  完結済みなので**発展枠**。
- **C5 / 4-R8 OCRマスキング（M〜L）**: tesseract.js で画像内テキストを検出し、
  メール/電話番号等を「モザイク候補」として提示→ワンクリック適用。重い処理のため
  バックグラウンド実行＋候補提示型 UX。
- **C9 / 5-4 描画性能（M）**: `render()` は全再描画方式。**先に計測**（項目 100/500/1000）し、
  体感遅延が出てから部分再描画化する。**問題が出ていない段階で最適化しない。**
