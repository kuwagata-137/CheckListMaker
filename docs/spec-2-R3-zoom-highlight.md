# 2-R3 要素矩形ベースの自動ズーム＋枠ハイライト — 確定仕様

ロードマップ **2-R3** の実装前に確定した仕様。2-R2 で取得できるようになった
**UI 要素の矩形**を使い、①クリック箇所の**拡大画像を自動生成**して「全景＋拡大」の
2枚構成にし、②全景のマーカーを「点（赤丸）」から**要素を囲む枠**へ進化させる。
矩形が取れないケースは常にフォールバックがあり、**必ず何かは出せる**（前提10）。

## ユーザー確定事項（2026-07-13）

1. **全景のマーカーは「枠」に進化させる。** UIA で要素矩形が採用できたときは全景に
   要素を囲む赤い枠を描き、採用できないときは従来どおりの赤丸にフォールバックする。
   ガジェットの既存トグルは「ハイライト全体（枠／赤丸）のオン/オフ」として引き続き
   機能する（表記は「クリック位置を赤で強調する」へ更新）。
2. **ステップ番号バッジは焼き込まない。** R4 取り込みウィザードで並べ替え・削除を
   したとき、画像に焼いた番号と実際の順番がズレるため。番号はサイドカーの連番のまま
   とし、表示は R4 以降の UI・手順書側が担う。
3. **拡大画像は撮影のたびに生成する**（録画停止時の一括や R4 時ではなく）。
   録画停止した時点でセッションフォルダに素材が完成している。生成は既存の保存
   直列キュー（persistChain）内で行うため録画は妨げない。

## 出力（セッションフォルダ）

```
<リスト名>_<yyyymmdd>_<hhmmss>/
  session.json
  001.png      ← 全景（ハイライト焼き込みは既存トグルに従う）
  001z.png     ← 拡大（全景から切り出し。ハイライトは全景と同じものが写る）
  001.json     ← サイドカー（スキーマ v3）
  ...
```

- 拡大は**ハイライト合成後の全景から切り出す**。1回の合成で全景と拡大の見た目が
  必ず一致し、処理も1系統で済む（トグル OFF なら素の画像から切り出すだけ）。
- 切り出しは `nativeImage.crop()`（メインプロセス内・レンダラー不要）。
- **拡大の生成失敗は撮影を止めない**（console.error のみ・サイドカーの `zoom` は
  null）。全景 PNG が主成果物である方針（2-R1 エラー方針）を維持する。

## 要素矩形の採用基準（枠＋要素ズームに使ってよい条件）

UIA が解決できても、矩形がハイライトとして意味を持たない場合がある。次を
**すべて**満たすときだけ採用し、満たさなければクリック座標フォールバックへ倒す。

1. `uia.resolved` かつ `rect` が有効（幅・高さ > 0）。
2. **コンテナ系（Window / Pane / Document / TitleBar）ではない**
   （steptext.js の `CONTAINER_TYPES` と同一の集合。RemoteApp 業務システムは
   ウィンドウ全体の矩形しか取れず、枠にしても情報がない——2-R0 発見①）。
3. **クリック座標が矩形の内側**（許容誤差 4px）。ElementFromPoint の結果が
   クリック点とズレている異常値を弾く。
4. **撮影画像とみなせる範囲に収まる**: 画像座標系に変換・クランプした結果、
   幅・高さがともに残っている。
5. **画面ほぼ全体ではない**: 幅・高さの**両方**が画像の 90% 以上なら不採用
   （枠にもズームにもならないため）。片方だけ大きい要素（横いっぱいのツールバー等）
   は正当なので採用する。

数値（4px・90%）は初期値であり、実機運用で調整してよい。

## 拡大画像の切り出し範囲

すべて物理px（撮影画像の座標系）。定数は `scaleFactor` 倍して使う（赤丸半径と同じ規約）。

- **要素採用時**: 要素矩形の四方に余白 `ZOOM_MARGIN = 48` を足した範囲。
  さらに最小サイズ `ZOOM_MIN_W × ZOOM_MIN_H = 480 × 320` を中心維持で保証する
  （チェックボックスのような小さい要素でも文脈が写る）。
- **フォールバック時**（矩形不採用）: クリック座標を中心に `480 × 320` 固定。
- **案A の広げ（2026-07-16 追加）**: 上記で決めた範囲（クランプ後）を、中心維持で
  `ZOOM_WIDEN = 1.5` 倍に広げてから再クランプする。要素採用時・フォールバック時の
  **両方**に効かせる。狭い切り出し（例 480×320）が表示側で列幅いっぱいに引き伸ばされ
  「拡大しすぎ」に見えていた問題への対応。比較モックで案A（現状の1.5倍広い）が
  もっともバランス良いと判断（見かけ倍率 約2.7× → 約1.8×）。
- どちらも最後に画像境界へクランプ（はみ出し分は内側へシフトし、それでも
  収まらなければ切り詰め）。画像が小さければ全景と同じになり得るが、それでよい。
- **拡大率は付けない**（切り出しのみ・アップスケールしない）。画素を捏造せず、
  表示側が幅に合わせて拡大表示することで「拡大」になる。

## ハイライトの描画

- **枠（要素採用時）**: 要素矩形を `FRAME_PAD = 6` 外側に広げた矩形を、既存の
  赤丸と同じ色 `#ef4444`・線幅 4（scaleFactor 比例）で描く。角丸半径 6。
- **赤丸（フォールバック時）**: 従来どおり半径 20・線幅 4 の中抜きリング。
- トグル OFF なら何も描かない（拡大画像の生成は行う）。
- 合成は既存 `drawMarker` の canvas 経路を拡張（circle / rect の描き分け）。
  4秒タイムアウト・失敗時は素の画像で続行、の既存方針は変えない。

## サイドカー（スキーマ v3）の変更点

```json
{
  "version": 3,
  "marker": {
    "drawn": true,
    "shape": "rect",
    "rect": [1180, 630, 108, 44],
    "lineWidth": 4,
    "color": "#ef4444"
  },
  "zoom": {
    "image": "001z.png",
    "rect": [1090, 542, 480, 320],
    "source": "element"
  }
}
```

- `marker.shape`: `"rect"`（枠）| `"circle"`（赤丸）。`rect` 形は描いた枠の矩形
  （FRAME_PAD 込み・画像座標）を `rect` に持ち、`circle` 形は従来どおり
  `x`/`y`/`radius` を持つ。`drawn: false`（トグル OFF）は形なし。
- `zoom`: 拡大画像のファイル名・切り出し範囲（画像座標）・根拠
  （`"element"`=要素矩形 / `"click"`=クリック中心フォールバック）。
  生成失敗・スキップ時は `null`。
- v2 からの互換: 既存フィールドの意味は変えない（R4 は `version` で分岐できる）。

## 実装配置

- **`zoomcrop.js`（新規・純関数）**: 矩形の座標変換（画面物理px→画像座標）、
  採用基準の判定、枠矩形と切り出し範囲の計算。fs・Electron 非依存で全分岐を
  単体テストする。
- **`steptext.js`**: `CONTAINER_TYPES` を export に追加（zoomcrop と判定を共有）。
- **`session.js`**: `SIDECAR_VERSION = 3`。`recordShot(png, meta)` の `meta.zoom =
  { png, rect, source }` を受けて `NNNz.png` を書き、サイドカーに `zoom` を記録
  （書き込み失敗は撮影を止めない）。
- **`main.js`**: `persistShot` で UIA 合流後に zoomcrop で計画を立て、
  `drawMarker`（circle/rect 対応に拡張）→ `nativeImage.crop()` で拡大を生成。
- **`gadget.html`**: トグル表記を「クリック位置を赤で強調する」へ変更（動作は同じ）。
- **`package.json`**: `build.files` に `zoomcrop.js` を追加。
- **テスト**: `test/zoomcrop.test.js`（新規）＋ `test/session.test.js`（v3 対応）。

## 検証計画

- 単体テスト: 採用基準の全分岐・切り出し範囲の境界（最小サイズ・画像端クランプ・
  巨大要素の不採用）・サイドカー v3 の記録内容。
- 実機検証（ユーザーの Windows 実機）: 録画して数クリック →
  ①全景に要素枠が描かれること（取れない対象では赤丸に戻ること）、
  ②`NNNz.png` がクリック箇所の読める拡大になっていること、
  ③トグル OFF で枠も丸も消え、拡大は生成されること。
  手順の詳細は実装完了時に報告する。

## スコープ境界（この工程でやらないこと）

- ステップ番号バッジの焼き込み（ユーザー確定で不採用。番号表示は R4 以降の UI が担う）。
- 取り込みウィザードでの拡大画像の利用・再生成（→ 2-R4）。
- ダブルクリック等の操作認識（→ 2-R2b）。
