# 2-R0 UIA検証スパイク 実測結果と設計への反映

- 実施日: 2026-07-13（ユーザーの Windows 実機・koffi 版 measure.js）
- 記録: 47クリック（生データはプライバシーのためリポジトリに含めない。
  社内システムのウィンドウタイトル等を含むため、本書は固有名を伏せた要約のみ）

## 結果サマリ

全体の「要素名あり」率: **44/47 = 94%**。ただしカテゴリごとに中身の質が異なる。

| カテゴリ（実測対象） | 取得率 | 解決時間(中央値) | 質の評価 |
| --- | --- | --- | --- |
| Windows標準（エクスプローラー・タスクバー・スタート等） | 100% | 10〜30ms | 「スタート」等、実用的な名前がそのまま取れる。理想的 |
| Electron アプリ | 78% | 39ms | ボタンの表示文言まで取れる。無名2件は空領域クリック（正当なフォールバック対象） |
| Chrome（ブラウザ自体のUI） | 75% | 62ms | タブ・ブックマーク等は取れる。ページ内コンテンツの深い解決は1件失敗（要追加検証） |
| リモートデスクトップ(RemoteApp)上の業務システム | 数字上100% | 5〜40ms | **実質はウィンドウタイトルのみ**（下記の発見①） |
| ネイティブエディタ（独自描画系） | 1件のみ | 188ms | 名前がクラス名相当で実用性は低い。フォールバック対象 |

※ Office系（Excel/Word/Outlook）は未計測。手順書の対象に入るなら後日追加計測する
（summarize.js で結果を合算できる）。

## 発見と設計への反映

1. **RemoteApp（mstsc 経由）の業務システムは要素単位の解決ができない。**
   UI 要素の実体はリモートサーバー側にあり、クライアント側 UIA から見えるのは
   RAIL ウィンドウ（クラス `RAIL_WINDOW`、種類 Window/Pane）まで。取れる「名前」は
   ウィンドウタイトルに等しい。
   → **R2 のフォールバック文生成（「ウィンドウ『◯◯』内の図の位置をクリック」）が、
   この種の業務アプリでは主系統になる**（前提10 の実証）。幸いダイアログ単位で
   タイトルが変わるため、フォールバック文でも画面の区別がつく。
   → R3 の自動ズームも、この場合は要素矩形が「ウィンドウ全体」になるため
   **クリック座標中心の固定サイズ切り出し**が主系統になる。
2. **koffi(FFI)＋UIA COM の本番経路は実機で成立**（selftest → 本計測まで完走）。
   解決時間は中央値 5〜60ms・最大188ms で、mousedown と並行の非同期解決（R2 設計）で
   完全に隠せる。C++ アドオンへの昇格は現時点で不要。
3. **Windows標準・XAML/Win32・Electron(Chromium) は名前の質が高い。**
   テンプレート文生成（R2）の主対象として計画どおり進められる。
4. **Chrome のページ内コンテンツ解決は追加検証が必要。**
   ブラウザ自体のUIは取れるが、ページ内の深い要素解決に失敗した例が1件。
   ブラウザ内業務システムを本格対象にする際は、アクセシビリティ有効化条件
   （UIA クライアント接続で自動有効化 / `--force-renderer-accessibility`）を
   R2 実装時に再計測して確かめる。
5. **ダブルクリックは連続クリックのペア（約200ms間隔）として観測された。**
   uiohook の `clicks` フィールド（計測ツールにも記録済み）で判定できる。

## 判定（go/no-go）

**go — 2-R2 は計画どおり進める。**
Windows標準 100%（基準70%以上）・ブラウザ 75%（基準50%以上）を満たす。
独自業務アプリ（RemoteApp）はフォールバック主系統として設計する（想定内・前提10）。

## 確定事項（2026-07-13）

- **ダブルクリックの認識と「『◯◯』をダブルクリック」文の生成は必須要件**とする
  （ユーザー確定）。実装タイミングは計画どおり 2-R2b（R2 コア完成後）。
- 現行録画のデバウンス（連打を1枚に集約）は v2 でも撮影面では維持し、
  操作認識（クリック数）は撮影とは独立に記録する方針で R2b を設計する。
