# 2-R2b 操作種類の拡張（ダブルクリック・キーボード・ドラッグ・アプリ切替） — 確定仕様

ロードマップ **2-R2b** の実装前に確定した仕様。録画エンジン v2 の操作認識を
「左右クリック」から**ダブルクリック・文字入力・Enter/ショートカット・ドラッグ・
前面アプリの切り替え**へ広げる。R2 コア（UIA解決＋文生成）・R3（ズーム/枠）・
R1（サイドカー）の上に載る拡張であり、既存のクリック録画の挙動は変えない。

## ユーザー確定事項（2026-07-13）

1. **実装範囲は①〜⑤すべて**（①ダブルクリック［必須要件］、②文字入力検出、
   ③Enter・ショートカット検出、④ドラッグ記録［オプション］、⑤アプリ切替検出）。
2. **キーボード系ステップ（②③）にも画像を付ける。撮影タイミングは「入力確定時」**
   （②は入力の区切り＝次のクリック / Enter / ショートカット / 録画停止の時点。
   ③はキーを押した時点）。事前キャプチャの仕組みを流用し「操作の直前の画面」を使う。
3. **入力内容（押された文字）は一切記録しない。** 記録するのは「入力した」という
   事実のみ。内容記録のオプトイン UI は今回は実装しない（要望が出たら後続で）。
   ※ ショートカットの組み合わせ（"Ctrl+S" 等）は操作名であり内容ではないので記録する。
   打鍵数も記録しない（パスワード長が漏れるため）。
4. **⑤アプリ切替は今回サイドカーへの記録のみ。** 取り込みウィザードでの
   セクション自動分割は後続で実装する（2-R1 の「取り込みは後から何度でもやり直せる」
   設計により、記録さえあれば過去の録画にも将来適用できる）。

## サイドカー スキーマ v4

`NNN.json` に **`kind`**（操作種類）を追加する。v3 まではすべてクリックだったため、
`kind` 欠落（旧バージョン）は `"click"` とみなす（後方互換・読み取り側で補完）。

| kind | 操作 | 画像 | 追加フィールド |
| --- | --- | --- | --- |
| `click` | 左/右クリック・ダブルクリック（従来＋①） | 従来どおり（全景＋拡大） | （従来どおり `click.clicks` に実クリック数） |
| `input` | 文字入力の確定（②） | 確定時点の全景（＋フォーカス要素の枠/拡大） | `keys: { type:"input", enter:bool }` |
| `key` | Enter 単独・ショートカット（③） | 押下時点の全景（枠/拡大なし） | `keys: { type:"shortcut", combo:"Ctrl+S" }` |
| `drag` | ドラッグ（④） | 始点=`NNN.png`・終点=`NNNe.png`（拡大なし） | `drag: { from:{x,y}, to:{x,y}, endImage, endImagePoint, endUia }` |

- 共通: `appChange: { from, to } | null` を追加（⑤。下記）。
- `input` の `uia` は**フォーカス要素**（クリック座標ではなく `GetFocusedElement` で解決）。
  `click` の各値は null。`imagePoint` は要素矩形の中心（矩形が取れないときは null・
  マーカーなし）。矩形が取れれば R3 と同じ枠ハイライト＋拡大 `NNNz.png` を生成する。
- `key` はマーカー・拡大なし（対象矩形が定義できない）。`click` の各値は null。
- `drag` の `uia` は**始点**の解決結果、`endUia` は**終点**。マーカーは始点・終点画像の
  それぞれに従来の赤丸（`marker` は始点分を記録、終点分は `drag.endMarker`）。拡大なし。

## ① ダブルクリックの認識（必須要件）

**現状**: 撮影デバウンス（300ms＋24px）が2回目のクリックを1枚に集約するため、
サイドカーは `clicks: 1` のまま＝ダブルクリックが記録に残らない。

**手法**: 撮影は従来どおり1枚に集約し、**操作認識だけ後から更新する**。
mouseup がデバウンスで捨てられたとき、`e.clicks >= 2` かつ直前保存ステップと
近接（デバウンスと同じ判定）かつ左ボタンなら、**直前のサイドカーを修正**する
（`session.amendLastShot()`。保存キュー `persistChain` に積んで順序を保証）:

- `click.clicks` を実測値へ更新（2=ダブル、3以上もそのまま記録）。
- `text` を再生成: 「**◯◯**」**をダブルクリック**（種類別テンプレートは使わない。
  Excel セルは「セル『B5』をダブルクリック」、フォールバックは
  「ウィンドウ『◯◯』内の図の位置をダブルクリック」）。`clicks >= 2` は一律
  「ダブルクリック」の文言とする（トリプル以上は稀で、記録は clicks に残る）。
- 右ボタンのダブルクリックは対象外（右クリック文のまま）。

## ② 文字入力の検出 → 「『◯◯』欄に入力」

**タイピングバースト方式**: uiohook の `keydown` を監視し、**印字キー**
（文字・数字・記号・スペース・テンキー）で「入力中」状態を開始する。
Backspace/Delete/矢印等の編集キーは入力中の継続とみなす（開始はしない）。

- **開始時**: `uia.resolveFocused()`（新設。下記）を1回だけ非同期キック
  （入力を受けている要素＝フォーカス要素の名前・矩形を取る）。
- **確定（＝1ステップとして保存）のトリガー**: 次の mousedown（クリック処理より
  先に確定）／ Enter ／ ショートカット ／ 録画停止。アイドルタイムアウトは
  設けない（考え中の休止で切れると1入力が複数ステップに割れるため）。
- **撮影**: 確定時点の事前キャプチャ（≦1.5秒前のフレーム）を優先、無ければ
  その場で撮影。Enter 確定の場合も「Enter の効果が出る前」の画面が写る
  （クリックの事前キャプチャと同じ理屈。入力済みの内容が画面に写っている）。
  対象ディスプレイはフォーカス要素矩形の中心（無ければカーソル位置）で決める。
- **文**: 名前あり → 「**◯◯**」欄に入力（Enter 確定なら「『◯◯』欄に入力して
  Enter」）。フォールバック → ウィンドウ「**◯◯**」内で文字を入力（(して Enter)。
  タイトルも無ければ「文字を入力」）。
- **自アプリ除外**: 自分のウィンドウ（ガジェット・本体）にフォーカスがあるときは
  バーストを開始しない（`BrowserWindow.getFocusedWindow()` で判定。クリックの
  `isOnOwnWindow` と同じ趣旨）。

## ③ Enter・ショートカットの検出

- **対象**: Ctrl / Alt / Win を含む組み合わせ（modifier＋通常キー）、単独の
  Enter、単独のファンクションキー（F1〜F12）。Shift のみの組み合わせは文字入力
  （②）であり対象外。Esc・Tab・矢印などその他の単独キーは記録しない。
- **キーリピート抑止**: `keydown`/`keyup` で押下中キー集合を管理し、
  「押されていない→押された」の遷移だけ発火。同一コンボの連打は撮影デバウンスと
  同じ 300ms で集約する。
- 発火時にタイピングバースト（②）が生きていれば**先に②を確定**する
  （順序: 入力ステップ → ショートカットステップ）。Enter はバースト中なら
  ②の確定フラグになるだけで、独立の `key` ステップは作らない。
- **撮影**: 押下時点の事前キャプチャ優先（効果が出る前の画面）。マーカー・拡大なし。
- **文**: 既知の操作は名前で 「**Ctrl+S で保存**」（対応表: S=保存, C=コピー,
  V=貼り付け, X=切り取り, Z=元に戻す, Y=やり直し, P=印刷, F=検索, A=すべて選択,
  N=新規作成, O=開く, W=閉じる, Alt+F4=ウィンドウを閉じる, F5=更新, F2=名前の変更）。
  未知の組み合わせは 「**Ctrl+Shift+K** キーを押す」、単独 Enter は
  「**Enter** キーを押す」。
- キーコード→表示名の変換表は **`keys.js`（新規・純関数）** に自前で持つ
  （libuiohook の VC_ 定数値。uiohook-napi の `UiohookKey` と同値だが、テストを
  ネイティブモジュール非依存に保つため実行時 import はしない）。

## ④ ドラッグの記録（オプション・既定 OFF）

- **ガジェットに「ドラッグを記録する」チェックを追加**（既定 OFF。赤丸トグルと
  同じパターンで `rec:setDrag` IPC・セッション開始ごとに OFF へ戻す）。
- OFF: 従来どおりドラッグは撮影しない（現行の skip 挙動そのまま）。
- ON: mousedown→mouseup の移動が **24px（`DRAG_MIN_DIST`）超**の左ボタン操作を
  ドラッグとして1ステップ記録する（現行の `DRAG_THRESHOLD`=5px はクリック判定用の
  ままにし、5〜24px の中間はクリックでもドラッグでもない＝従来どおり捨てる。
  誤爆防止）。
  - 始点画像 = mousedown 時の事前キャプチャ（従来のクリック撮影と同じ経路）＋赤丸。
  - 終点画像 = mouseup 時にその場で撮影（`NNNe.png`）＋赤丸。
  - UIA は始点（mousedown で既にキック済み）と終点（mouseup でキック）の両方を解決。
  - **文**: 「**◯◯**」から「**△△**」へドラッグ。名前が取れない側は
    「図の始点位置」「図の終点位置」で補う（両方フォールバックなら
    「ウィンドウ『◯◯』内の図の始点から終点へドラッグ」）。
  - 拡大画像は生成しない（範囲操作に「1要素の拡大」は合わない）。

## ⑤ 前面アプリ切り替えの検出（サイドカー記録のみ）

- 保存する各ステップで、UIA 解決結果の `appName` を直前ステップと比較し、
  **両方が非 null で異なるとき** `appChange: { from, to }` をサイドカーに記録する
  （最初のステップ・解決できなかったステップは null）。
- ポーリングでの前面窓監視はしない（ステップの無い区間の切り替えは手順書の
  章立てに影響しないため、ステップ時点の比較で足りる）。
- 取り込みウィザードのセクション自動分割・「ここから◯◯での作業」の見出し生成は
  **後続実装**。本工程は将来それを可能にする記録まで。

## uia-host.js / uia.js の拡張: フォーカス要素の解決

- 親からのメッセージに `focus: true` を追加: `{id, focus:true}` なら
  `IUIAutomation::GetFocusedElement`（vtable index 8）で解決する
  （座標版と同じプロパティ一式を返す）。ウィンドウタイトル・アプリ名は
  `GetForegroundWindow`（Win32）から取る。
- `uia.js` に `resolveFocused()` を追加（タイムアウト・フォールバックは
  `resolve(x,y)` と同一方針。非 Windows は null）。

## 取り込みウィザード（2-R4）の追従（最小）

- `readSession` がステップへ `kind` / `keys` / `drag`（`endImage` 含む）/
  `appChange` を通す（旧サイドカーは `kind:"click"` 扱い）。
- `input` / `key` ステップは既存の画像選択ロジックにそのまま乗る
  （拡大があれば拡大、無ければ全景）。
- `drag` ステップは終点画像を「拡大」スロットに載せ、画像切替の表示だけ
  **始点／終点／両方**（既定: 両方・並びは 始点 → 終点）に読み替える。
- `rec:image` のファイル名検証を `NNNe.png` も許可する形へ更新。

## エラー方針（従来を踏襲）

- キーボード監視・フォーカス解決・終点撮影の失敗は録画を止めない
  （ステップがフォールバック文・画像なしになるだけ。console.error のみ）。
- `amendLastShot` の失敗は元のクリックステップを壊さない（修正を諦めるだけ）。
