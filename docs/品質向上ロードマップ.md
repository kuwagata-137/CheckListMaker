# CheckListMaker 品質向上ロードマップ

アプリのクオリティを、**既存のSaaS（Scribe / Tango 等）と比較しても戦えるレベル**へ
引き上げるための開発計画。戦略は一言でいうと——

> **「SaaS がブラウザとクラウドの中でやっていることを、デスクトップ全体とローカル完結でやる」**

SaaS の弱点（デスクトップアプリの操作に弱い・データが外部サーバーに出る・月額課金）の
裏側を突く。順序は「**データを失わない → 操作を理解する録画 → 使う体験 → 発展**」。

各項目は「**問題 → 手法 → 効果 → 規模感**」で書く。規模感の目安:
**S**=半日〜1日 / **M**=数日 / **L**=1〜2週間（検証込み）。

## 到達レベルの定義

本計画がどこを目指すかを先に定義する（2026-07-11 確定: **SaaS対抗ラインを目標とする**）。

| レベル | 内容 | 対応フェーズ |
| --- | --- | --- |
| 最低限ライン | 安心して使える完成された道具。土台修理＋既存コンセプトの完成 | フェーズ1 |
| **SaaS対抗ライン（目標）** | **録画したら、文章と拡大画像付きの手順書が出来ている。データは端末から一歩も出ない** | フェーズ2〜3 |
| 発展ライン | ローカルAI仕上げ・自動マスキングまで含め ―「AI込みで完全オフライン」 | フェーズ4 |
| 追わない領域 | クラウドであることが本質の機能（後述「追わないと決めた領域」） | — |

## 前提（確定済みの技術判断）

1. **レンダラーは単一HTML構成を維持する**（`index.html` 1ファイル・依存ゼロ・`file://` 動作）。
   ビルド工程（バンドラ）は導入しない。
2. **メインプロセス側のファイル分割は許容する**（2026-07-11 確定）。録画エンジン v2 で
   main.js が肥大するため、`recorder.js` / `uia.js` / `session.js` 等への分割を認める。
   レンダラーの単一HTML特性とは独立の決定。
3. **保存の移行先は「JSON＋画像個別ファイル」**。SQLite 等のネイティブDBは使わない。
4. **ブラウザで `index.html` 単体を開いても動く特性を現状同等に維持する**
   （Electron=ファイル保存 / ブラウザ単体=localStorage の2系統）。
5. **実施記録（点検ログ）は当面作らない**（「将来枠」に記載のみ）。
6. **macOS 対応は当面追わない**。録画エンジン v2 は Windows（UI Automation）に集中する。
   二正面作戦は両方を中途半端にする。

## フェーズ1: 信頼性の土台 — データを失わない・壊れない（R0）

機能を増やす前に構造的な弱点を塞ぐ。**どの到達レベルを目指すとしてもここが最初**。
フェーズ2以降でネイティブ連携（UIA・FFI）が増えるほど、テストと安定した保存基盤の
価値は上がる。

### 1-1. 保存基盤の移行（最優先） — 規模 L

**問題**

- 全データ（画像含む）を localStorage（実質約 5MB）に保存している（`index.html` の
  `LocalStorageAdapter`）。画像は長辺 1600px・JPEG 品質 0.85 の dataURL（1枚 150〜400KB
  程度）で state に直接埋め込まれるため、**写真を数十枚貼った時点で保存が失敗し始める**。
  しかも失敗は画面に出ない。
- Undo が「画像込みの全 state」を最大 50 世代フルクローンして保持する（`Store.commit`）。
  画像が増えるとメモリが数百MB規模に膨らみ得る。

**手法**

- **Electron 側（メインプロセス）**
  - `preload.js` に contextBridge で `window.storageAPI` を追加する。既存の
    `recorderAPI` と同じパターン（`contextIsolation: true` のまま安全に公開）。
  - `main.js` に IPC ハンドラを追加:
    `storage:load` / `storage:save` / `image:save` / `image:get` / `image:delete`。
  - 保存先はユーザーデータフォルダ:
    - 本体: `app.getPath('userData')/data/checklists.json`
    - 画像: `app.getPath('userData')/data/images/<uuid>.jpg`（1枚1ファイル）
  - 書き込みは**アトミック方式**（一時ファイルに書いて `fs.renameSync`）＋
    直前世代を `checklists.json.bak` として保持。書き込み途中のクラッシュでも
    直前状態に戻れる。
- **レンダラー側（index.html）**
  - 既存の `StorageAdapter` 抽象（`LocalStorageAdapter` / `MemoryAdapter` が実装済み）に
    **`FileAdapter` を追加**する。起動時に `window.storageAPI` の有無で自動選択:
    - Electron → `FileAdapter`（ファイル保存）
    - ブラウザ単体 → 従来どおり `LocalStorageAdapter`（**無改修で互換維持**）
    - 書き出した単体HTML → 従来どおり `MemoryAdapter`
  - 画像の持ち方を変更: `item.images` の dataURL を **`img:<uuid>` の参照**にする
    （Electron モードのみ）。表示時に `image:get` で解決し、Blob URL でキャッシュ。
    ブラウザモードは従来どおり dataURL 直持ち。
  - **入出力の互換を守る**: JSON エクスポート・単体HTML書き出し（`buildStandaloneHtml`）の
    際は参照を dataURL に戻して埋め込む。インポート時は dataURL を画像ファイルへ
    展開して参照化する。→ 既存のバックアップ・共有ファイルはそのまま使える。
- **マイグレーション**
  - Electron 初回起動時に localStorage の既存データ（`checklistmaker.v1`）を検出したら
    ファイルへ一括変換し、完了マーカーを書く。変換前の localStorage は消さない
    （切り戻し可能にする）。
- **Undo の軽量化**: 画像が参照（短い文字列）になるため 50 世代クローンは自然に軽くなる。
  `imageEdits`（注釈の元データ）も同様に参照化する。

**効果**: 容量上限の実質撤廃／データ消失リスクの解消／Undo のメモリ膨張解消／
原寸画像保存（3-3）と録画セッション形式（2-R1）の土台。

### 1-2. 自動テストと CI — 規模 M

**問題**: 5,130 行の単一 HTML に自動テストが 0 本。1-1 のような危険な変更を
目視確認だけで進めるのはリスクが高すぎる。

**手法**（単一HTML維持の制約下でのやり方）

- **テストハーネス方式**: `test/harness.js` が `index.html` を読み込み `<script>` 部分を
  抽出して jsdom 環境で評価する。アプリ側にはテスト時のみ内部関数を公開するフックを
  1箇所だけ足す（`window.__test__` が未定義なら何もしない。本番動作に影響なし）。
- テストランナーは **Node 標準の `node:test`**＋jsdom。依存を最小に保つ。
- **対象の優先順位**:
  1. 1-1 のマイグレーション処理（最重要）
  2. model: 項目/セクションの追加・移動・削除、進捗計算、一括リセット
  3. io: JSON入出力、単体HTML書き出し/取り込み、URL共有のエンコード/デコード
  4. フェーズ2以降: テンプレート文生成（2-R2）は純関数なので必須テスト対象
- **CI**: `.github/workflows/` にテスト用 workflow を追加（push / PR で実行）。

### 1-3. エラーの可視化 — 規模 S

**問題**: 実行時エラーや保存失敗が console 止まりで、誰にも見えない。

**手法**: `window.onerror` / `unhandledrejection` を捕捉して userData 配下の
ローカルログへ記録（**外部送信は一切しない**）。保存失敗（`Store.lastSaveOk` は既存）を
トースト通知に昇格。

## フェーズ2: 録画エンジン v2 — 「画面を撮る」から「操作を理解する」へ（R1〜R4）

**SaaS対抗の核。** 現在の録画は「座標を取ってスクショを撮る」だけで、その後の
文章書き・画像整形がすべて手作業として残る。Scribe / Tango との差はここで生まれている。
録画を**操作の意味を記録するエンジン**に作り替える。

### 2-R1. 録画セッション形式（画像＋メタデータ併記） — 規模 S【最初にやる基盤】

**問題**: 撮影情報（クリック座標・時刻・対象ディスプレイ）を取得しているのに
PNG 保存後に**捨てている**。後段の全機能がこの情報を必要とする。

**手法**: 1クリックごとに JSON サイドカーを併記する。

```
録画セッション/<リスト名_日時>/
  001.png
  001.json   ← 座標、UI要素名、種類、要素矩形、ウィンドウタイトル、アプリ名、時刻
  002.png ...
```

- `persistShot`（main.js）で PNG と同時に JSON を書くだけ。R2 以前は UIA 項目が
  空のまま座標・時刻・ウィンドウ情報だけでも先に保存を始める。
- **効果**: 取り込み処理を後から何度でもやり直せる（取り込みロジックを改良したら
  過去の録画も恩恵を受ける）／デバッグ容易／R2〜R4・フェーズ4が全部この上に載る。

### 2-R2. UI要素解決＋テンプレート文生成（本丸） — 規模 L【検証スパイク先行】

**問題**: 手順テキストを全部ユーザーが書いている。SaaS は「設定画面で『保存』を
クリック」と**自動生成**しており、ここが最大の体験差。

**手法**

- **UI Automation (UIA)**: Windows 標準 API の `ElementFromPoint` で、クリック座標から
  UI 要素を取得する——要素名（「保存」）、種類（button/tab/edit/checkbox/menuitem）、
  **要素の矩形**、親ウィンドウのタイトル、アプリ名。
- 呼び出し手段は **koffi（FFI・コンパイラ不要）**から始める。性能や安定性で不足したら
  C++ N-API アドオンへ昇格。mousedown 時に撮影と**並行して非同期解決**すれば
  レイテンシは隠せる（既存の mousedown→mouseup 二段設計と相性が良い）。
- **テンプレート文法による決定的生成**（AI不要・オフライン・幻覚なし）:

  | 取得した情報 | 生成される文 |
  | --- | --- |
  | button「保存」 | 「**保存**」ボタンをクリック |
  | tab「ホーム」 | 「**ホーム**」タブを選択 |
  | edit「ファイル名」＋入力検出 | 「**ファイル名**」に入力 |
  | checkbox「同意する」 | 「**同意する**」にチェック |
  | menuitem「印刷」 | メニューから「**印刷**」を選択 |
  | 要素名なし（フォールバック） | ウィンドウ「◯◯」内の図の位置をクリック |

- **操作種類の拡張**（uiohook はキーボードも監視できる）:
  - 文字入力の検出 →「『◯◯』欄に入力」ステップを生成。**入力内容そのものは既定で
    記録しない**（プライバシー原則。事実のみ記録。内容記録はオプトイン）。
  - Enter・ショートカット検出（「Ctrl+S で保存」を1ステップに）。
  - ドラッグの記録（現在は破棄）→「◯◯から△△へドラッグ」を始点終点2枚付きで
    記録するオプション。
  - **前面アプリの切り替え検出 → セクション自動区切り**（「ここから Excel での作業」。
    手順書の章立てが録画だけで生まれる）。
- **着手前に検証スパイク（1〜2日）を必ず行う**: 主要アプリ10種（Excel / Word /
  ブラウザ / エクスプローラー / 業務系デスクトップアプリ等）で UIA の要素名が
  どれだけ取れるかを実測する。**この結果がフォールバック設計と期待値を決める。**
  一部アプリ（ゲーム・独自描画）では取れないことを前提に、常にウィンドウタイトルへ
  フォールバックできる設計にする。

**効果**: 「録画したら文章付きの手順書が出来ている」が成立。SaaS の核心機能と
オフラインで並ぶ。

### 2-R3. 要素矩形ベースの自動ズーム＋枠ハイライト — 規模 M（R2の副産物）

**問題**: 現在のスクショは「全画面＋赤丸」のみ。細部が読めず、SaaS の
「クリック箇所を自動で拡大して見せる」品質に届いていない。

**手法**

- R2 で取れる**要素の矩形**を使い、要素周辺を適切な余白で切り出した**拡大画像を
  自動生成**する。全景＋拡大の2枚構成を既定に。矩形が取れない場合はクリック座標
  中心の固定サイズ切り出しへフォールバック（必ず何かは出せる）。
- マーカーを「点（赤丸）」から**要素を囲む枠**に進化させ、ステップ番号バッジも
  自動で焼き込む。マーカー合成の canvas 経由の仕組みは既存（`drawMarker`）で、
  描く図形が変わるだけ。

**効果**: 「どこを押すか一目で分かる」SaaS 品質のスクショを人手ゼロで生成。

### 2-R4. 取り込みウィザード — 規模 M

**問題**: R1〜R3 で「素材」は自動生成されるが、そのまま完成にはならない
（余計なクリック・順序の調整・文言の微修正が必ずある）。

**手法**: 録画停止時に取り込みウィザードを表示——生成されたステップ一覧
（自動生成文＋拡大画像）を、**並べ替え・不要ステップの削除・隣接ステップの統合・
文言の一括調整**してから取り込む UI。取り込み先は「開いているリストの新セクション
（録画日時付き）」を既定。項目生成・画像添付は既存の `createItem` / `addItemImage` /
`compressImage` を流用。

**効果**: 「自動生成 → 3分の手直し → 完成」の作業フローが完結する。

## フェーズ3: 使う体験と出力 — 「作る」の次の「使う」で差をつける（R5〜R6＋出力系）

### 3-R5. 実行モード（ステッププレイヤー）＋書き出しHTML同梱 — 規模 M

**問題**: 出力が「印刷物（紙・PDF・Word）」の発想のみ。画面で使う時代の
閲覧体験（SaaS のステップビューア）に相当するものがない。

**手法**

- 手順書を「1ステップ全画面表示・完了チェック・キーボードで次へ」で進める
  **実行モード**をレンダラーに追加。所要時間フィールドが既にあるので、
  **実測時間の記録と標準時間との比較**も表示する（教育・作業標準化に刺さる）。
- **書き出した自己完結HTMLにも実行モードを同梱**する。手順書を1ファイル渡せば、
  受け取った人はブラウザだけでステップ実行できる（サーバー不要の共有として完成）。

### 3-R6. ガイドオーバーレイ — 規模 M【独自機能・SaaSに原理的に不可能】

**問題**: 手順書を見ながら作業するとき、ウィンドウを行き来する必要がある。
ブラウザ拡張型の SaaS はブラウザの外に出られない。

**手法**: 録画ガジェット（常時最前面・撮影に写らない小窓、`gadget.html`）の技術を
**そのまま「再生用ガジェット」に転用**する。実際の業務アプリを操作しながら、
画面の隅に「現在のステップ（拡大画像＋文）＋次へ/戻るボタン」が浮いている状態を作る。

**効果**: デスクトップ全体の上にガイドを浮かべられるのはネイティブアプリだけ。
既存資産の再利用で、明確な独自機能になる。

### 3-A. Excel / CSV 出力 — 規模 M

**問題**: 日本の現場の点検表・チェック記録は Excel 文化。Word 出力だけでは
提出フローに乗らない。

**手法**: **exceljs**（純JS・ネイティブ依存なし）で「No / 項目 / チェック / 所要時間 /
メモ」列＋セクション見出し行の .xlsx を生成。CSV は **BOM 付き UTF-8**
（Excel で文字化けしない）。保存 IPC は既存 `docx:save` と同型。ブラウザモードは
既存 `downloadHtml` と同じ Blob ダウンロードで対応。

### 3-B. 画像画質の向上 — 規模 S〜M（1-1 完了が前提）

**問題**: 容量制約のため取り込み時に長辺 1600px へ不可逆圧縮しており、細かい文字の
スクショが印刷・Word で潰れる。

**手法**: 1-1 完了後、**原寸の元画像を保存**し、画面表示用の縮小サムネイルを別に持つ
2段構成へ。印刷/Word/PDF 出力は原寸を使う。`IMG_MAX_EDGE` / `IMG_QUALITY` は
「表示用」の設定として残す。

### 3-C. Word 出力の依存リスク対応（調査タスク） — 規模 S

**問題**: `html-to-docx` はメンテナンスが停滞気味。主要機能が塩漬けライブラリに
依存している。

**手法**: 現状の出力サンプル一式をゴールデンファイルとして保存した上で、
活発にメンテされている **`docx`** パッケージへの移行コストと得失を1ドキュメントに
まとめる（後処理 `docx-postprocess.js` のハックは `docx` なら生成時に直接表現できる
見込み）。調査までをタスクとし、書き換えの判断は結果を見てから。

## フェーズ4: 発展モジュール — 「AI込みで完全オフライン」（R7〜R8）

### 4-R7. ローカルLLMによる文章仕上げ（任意モジュール） — 規模 L

**問題**: テンプレート生成文（2-R2）は正確だが、「クリック」の連続で文章としては
機械的。SaaS は生成AIでタイトル・要約・自然な文章まで作る。

**手法**

- **node-llama-cpp ＋ 小型日本語対応モデル（数GB）**を**任意ダウンロードの
  オプションモジュール**として統合。既定は無効（本体の軽さを守る）。
- 用途は絞る: ①テンプレート文の自然文化・丁寧語化、②手順全体からのタイトル・概要・
  注意書きの下書き生成、③「新人向けに言葉を柔らかく」等の一括リライト。
- **推論も完全ローカル。データは1バイトも外に出ない。**「AI仕上げまで含めて
  オフライン」はクラウドSaaSに原理的に真似できない売り文句になる。

### 4-R8. 機密情報マスキング支援 — 規模 M〜L

**問題**: 機密のぼかしが手動モザイクのみ。SaaS は自動でぼかす。

**手法**: **tesseract.js（ローカルOCR・日本語対応）**で画像内テキストを検出し、
メールアドレス・電話番号等のパターン一致領域を「モザイク候補」として提示、
ワンクリック適用。処理が重いためバックグラウンド実行＋候補提示型のUXにする。

## フェーズ5: 使い込みに耐える — スケールと操作性

リストが増え、日常的に使われる段階で効く改善。**フェーズ2〜4の合間に S 規模のものから
差し込んでよい**（依存が薄いため）。

- **5-1. 全文検索（S）**: ホームに検索ボックス。タイトル・項目・メモを横断する
  インクリメンタル検索。データは全件メモリ上にあるため**索引不要の線形走査で十分**
  （数百リスト規模まで）。ヒットのハイライトとエディタ内ジャンプまでやると完成度が高い。
- **5-2. ホーム整理: タグ＋アーカイブ（M）**: model に `tags: string[]` と
  `archived: bool` を追加（旧データは `Store` 初期化時のデフォルト補完——`settings`
  補完と同じ既存パターン）。フォルダ階層よりタグが model 変更が小さく D&D 移動も
  不要なので、先にタグで様子を見る。

## 追わないと決めた領域（Tier C）

## 技術リスク（継続的に把握しておくこと）

| リスク | 内容と対応方針 |
| --- | --- |
| UIA の当たり外れ | アプリによって要素名が取れない（ゲーム・独自描画）。**R2 着手前の検証スパイクで実測**し、ウィンドウタイトルへのフォールバックを常に用意する。 |
| koffi（FFI）の安定性 | クラッシュはメインプロセスを巻き込む。UIA 呼び出しは try/catch と時間制限で包み、失敗時は「要素情報なし」で続行。不足なら C++ N-API アドオンへ昇格。 |
| ローカルLLM のサイズと品質 | 数GBのモデル配布・小型モデルの日本語品質は要検証。任意モジュール（既定無効）とし、テンプレート生成（R2）だけで実用になる設計を崩さない。 |
| `html-to-docx` のメンテ停滞 | Word 出力の土台。3-C の調査で移行パスを確保しておく。 |
| Electron の追随更新 | 現在 31 系。セキュリティ修正が頻繁なため定期的なメジャー追随を運用に組み込む（テスト整備後はコストが下がる）。 |
| 単一 HTML の行数増加 | レンダラーは方針として維持。既存のセクション構成（model/storage/io/render/…）と関数粒度の規律で対処。**メインプロセス側は分割を許容**（前提2）。 |
| `uiohook-napi` の AV 誤検知 | グローバル入力フックは疑われやすい。挙動（クリック座標のみ・外部送信なし）を README/ガイドに明記。 |

## 実施順序のまとめ

```
R0 フェーズ1（土台）  : 1-1 保存移行 → (1-2 テスト+CI は並行可) → 1-3 エラー可視化
   フェーズ2（録画v2）: R2検証スパイク(1〜2日) → R1 セッション形式 → R2 UIA+文生成
                        → R3 自動ズーム/枠 → R4 取り込みウィザード
   フェーズ3（使う）  : R5 実行モード+HTML同梱 → R6 ガイドオーバーレイ
                        → 3-A Excel/CSV → 3-B 画像画質 → 3-C docx調査
   フェーズ4（発展）  : R7 ローカルLLM → R8 OCRマスキング
   フェーズ5（操作性）: 5-1 検索 → 5-2 タグ → 5-3 キーボード/a11y → 5-4 性能(計測後)
                        ※ S規模のものはフェーズ2〜4の合間に差し込み可
```

- **R1→R2→R3 が完了した時点で「録画したら、文章と拡大画像付きの手順書が出来ている。
  データは端末から一歩も出ない」が成立する。** これが本計画の中間ゴール
  （SaaS対抗ライン）であり、比較表で唯一負けるのが「クラウド共有」だけの状態。
- 最初の一歩は **R2 の検証スパイク**（主要アプリ10種で UIA 要素名の取得率を実測）。
  結果が R2/R3 の設計と期待値を決めるため、本実装より先に行う。
- 各フェーズ末で前フェーズの回帰テストを回す。
