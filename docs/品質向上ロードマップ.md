# CheckListMaker 品質向上ロードマップ

アプリのクオリティを、**既存のSaaS（Scribe / Tango 等）と比較しても戦えるレベル**へ
引き上げるための開発計画。戦略は一言でいうと——

> **「SaaS がブラウザとクラウドの中でやっていることを、デスクトップ全体とローカル完結でやる」**

SaaS の弱点（デスクトップアプリの操作に弱い・データが外部サーバーに出る・月額課金）の
裏側を突く。順序は「**データを失わない → 操作を理解する録画 → 使う体験 → 発展**」。

各項目は「**問題 → 手法 → 効果 → 規模感**」で書く。規模感の目安:
**S**=半日〜1日 / **M**=数日 / **L**=1〜2週間（検証込み）。

> **残作業の一覧は `残りの開発計画.md` に集約している**（2026-07-16 取りまとめ）。
> 「いま何が残っているか」を素早く知りたいときはそちらを見る。本書は各項目の
> 背景・手法・確定仕様までを含む一次資料として維持する。

## 到達レベルの定義

本計画がどこを目指すかを先に定義する（2026-07-11 確定: **SaaS対抗ラインを目標とする**）。

| レベル | 内容 | 対応フェーズ |
| --- | --- | --- |
| 最低限ライン | 安心して使える完成された道具。土台修理＋既存コンセプトの完成 | フェーズ1 |
| **SaaS対抗ライン（目標）** | **録画したら、文章と拡大画像付きの手順書が出来ている。データは端末から一歩も出ない** | フェーズ2（R1〜R4）〜3 |
| 発展ライン | ローカルAI仕上げ・自動マスキングまで含め ―「AI込みで完全オフライン」 | フェーズ4 |
| 追わない領域 | クラウドであることが本質の機能（後述「追わないと決めた領域」） | — |

### 中間ゴールの受け入れ基準（測定可能な指標）

SaaS対抗ラインは **R1〜R4 の完了**（取り込みまで）で成立とする（2026-07-11 確定）。
「出来ている」を主観にしないため、次の指標で測る。数値はすべて**初期値**であり、
2-R0 スパイクの実測と運用で調整してよい。

- **手直し時間**: 録画停止 → 取り込み完了までの編集作業が、10手順程度の記録で **5分以内**。
- **自動生成文の採用率**: スパイク対象アプリでの操作について、無修正のまま使える生成文が
  **7割以上**（ウィンドウタイトルへのフォールバック文を含む）。
- **画像品質**: 全ステップで「全景＋クリック箇所が判読できる拡大」の2枚が人手ゼロで揃う。

## 前提（確定済みの技術判断）

1. **レンダラーは単一HTML構成を維持する**（`index.html` 1ファイル・依存ゼロ・`file://` 動作）。
   ビルド工程（バンドラ）は導入しない。
2. **メインプロセス側のファイル分割は許容する**（2026-07-11 確定）。録画エンジン v2 で
   main.js が肥大するため、`recorder.js` / `uia.js` / `session.js` 等への分割を認める。
   レンダラーの単一HTML特性とは独立の決定。
3. **保存の移行先は「JSON＋画像個別ファイル」**。SQLite 等のネイティブDBは使わない。
4. **ブラウザで `index.html` 単体を開いても動く特性を現状同等に維持する**
   （Electron=ファイル保存 / ブラウザ単体=localStorage の2系統）。
5. **実施記録（点検ログ）は当面作らない**（「将来枠」に記載のみ）。
6. **macOS 対応は当面追わない**。録画エンジン v2 は Windows（UI Automation）に集中する。
   二正面作戦は両方を中途半端にする。
7. **品質基準は配布先で変えない**（2026-07-11 確定）。「既存SaaSと戦える」は一般配布に
   耐える水準を指す。配布レディ化（コード署名・自動更新・リリース工程）を計画に含める
   （→ フェーズ3.5）。
8. **中間ゴール（SaaS対抗ライン）の成立点は R1〜R4 完了**（2026-07-11 確定）。
   素材がセッションフォルダに出来るだけでなく、取り込み（R4）まで済んで初めて
   「録画したら手順書が出来ている」と言える。
9. **UIA 検証スパイクは計画全体の最初に実施する**（2026-07-11 確定。フェーズ1より先）。
   Claude が計測スクリプトを用意し、ユーザーが Windows 実機で実行する（→ 2-R0）。
10. **スパイク対象は4カテゴリすべて**（2026-07-11 確定）: Office系 / ブラウザ内の業務
    システム / 独自の業務デスクトップアプリ / Windows標準。独自アプリを含むため、
    フォールバック設計を「例外処理」ではなく主系統の一つとして扱う。

## フェーズ1: 信頼性の土台 — データを失わない・壊れない（R0）

機能を増やす前に構造的な弱点を塞ぐ。**実装作業としてはここが最初**（計画全体の最初は
2-R0 検証スパイクだが、その実測はユーザーの Windows 実機作業のため、待ち時間に本フェーズを
並行して進める）。フェーズ2以降でネイティブ連携（UIA・FFI）が増えるほど、テストと安定した
保存基盤の価値は上がる。

### 1-1. 保存基盤の移行（最優先） — 規模 L

**問題**

- 全データ（画像含む）を localStorage（実質約 5MB）に保存している（`index.html` の
  `LocalStorageAdapter`）。画像は長辺 1600px・JPEG 品質 0.85 の dataURL（1枚 150〜400KB
  程度）で state に直接埋め込まれるため、**写真を数十枚貼った時点で保存が失敗し始める**。
  しかも失敗は画面に出ない。
- Undo が「画像込みの全 state」を最大 50 世代フルクローンして保持する（`Store.commit`）。
  画像が増えるとメモリが数百MB規模に膨らみ得る。

**手法**

- **Electron 側（メインプロセス）**
  - `preload.js` に contextBridge で `window.storageAPI` を追加する。既存の
    `recorderAPI` と同じパターン（`contextIsolation: true` のまま安全に公開）。
  - `main.js` に IPC ハンドラを追加:
    `storage:load` / `storage:save` / `image:save` / `image:get` / `image:delete`。
  - 保存先はユーザーデータフォルダ:
    - 本体: `app.getPath('userData')/data/checklists.json`
    - 画像: `app.getPath('userData')/data/images/<uuid>.jpg`（1枚1ファイル）
  - 書き込みは**アトミック方式**（一時ファイルに書いて `fs.renameSync`）＋
    直前世代を `checklists.json.bak` として保持。書き込み途中のクラッシュでも
    直前状態に戻れる。
- **レンダラー側（index.html）**
  - 既存の `StorageAdapter` 抽象（`LocalStorageAdapter` / `MemoryAdapter` が実装済み）に
    **`FileAdapter` を追加**する。起動時に `window.storageAPI` の有無で自動選択:
    - Electron → `FileAdapter`（ファイル保存）
    - ブラウザ単体 → 従来どおり `LocalStorageAdapter`（**無改修で互換維持**）
    - 書き出した単体HTML → 従来どおり `MemoryAdapter`
  - 画像の持ち方を変更: `item.images` の dataURL を **`img:<uuid>` の参照**にする
    （Electron モードのみ）。表示時に `image:get` で解決し、Blob URL でキャッシュ。
    ブラウザモードは従来どおり dataURL 直持ち。
  - **入出力の互換を守る**: JSON エクスポート・単体HTML書き出し（`buildStandaloneHtml`）の
    際は参照を dataURL に戻して埋め込む。インポート時は dataURL を画像ファイルへ
    展開して参照化する。→ 既存のバックアップ・共有ファイルはそのまま使える。
- **マイグレーション**
  - Electron 初回起動時に localStorage の既存データ（`checklistmaker.v1`）を検出したら
    ファイルへ一括変換し、完了マーカーを書く。変換前の localStorage は消さない
    （切り戻し可能にする）。
- **Undo の軽量化**: 画像が参照（短い文字列）になるため 50 世代クローンは自然に軽くなる。
  `imageEdits`（注釈の元データ）も同様に参照化する。

**効果**: 容量上限の実質撤廃／データ消失リスクの解消／Undo のメモリ膨張解消／
原寸画像保存（3-B）と録画セッション形式（2-R1）の土台。

### 1-2. 自動テストと CI — 規模 M【✅ 実施済み 2026-07-13】

> **結果**: `test/`（node:test＋jsdom、31テスト）と `.github/workflows/test.yml` を追加。
> 対象は 1-1 マイグレーション（統合）・保存基盤（storage.js 単体）・model 純関数・
> 共有リンク/単体HTML の io。確定した実装方針は `docs/spec-1-2-test-ci.md` 参照
>（原案の「script 抽出評価」ではなく jsdom で index.html を丸ごと起動する方式に変更）。

**問題**: 5,130 行の単一 HTML に自動テストが 0 本。1-1 のような危険な変更を
目視確認だけで進めるのはリスクが高すぎる。

**手法**（単一HTML維持の制約下でのやり方）

- **テストハーネス方式**: `test/harness.js` が `index.html` を読み込み `<script>` 部分を
  抽出して jsdom 環境で評価する。アプリ側にはテスト時のみ内部関数を公開するフックを
  1箇所だけ足す（`window.__test__` が未定義なら何もしない。本番動作に影響なし）。
- テストランナーは **Node 標準の `node:test`**＋jsdom。依存を最小に保つ。
- **対象の優先順位**:
  1. 1-1 のマイグレーション処理（最重要）
  2. model: 項目/セクションの追加・移動・削除、進捗計算、一括リセット
  3. io: JSON入出力、単体HTML書き出し/取り込み、URL共有のエンコード/デコード
  4. フェーズ2以降: テンプレート文生成（2-R2）は純関数なので必須テスト対象
- **CI**: `.github/workflows/` にテスト用 workflow を追加（push / PR で実行）。

### 1-3. エラーの可視化 — 規模 S【✅ 実施済み 2026-07-13】

> **結果**: `errorlog.js`（`<userData>/logs/error.log` へ JSONL・512KBローテーション・
> 外部送信なし）とレンダラーのグローバル捕捉、保存失敗の右下トースト
>（`notifySaveError` に一本化・30秒スロットル）を実装。確定仕様は
> `docs/spec-1-3-error-visibility.md`、テストは `test/errorlog.test.js` /
> `test/errors.test.js`（計9件追加・全40件）。**これでフェーズ1（土台）完了。**

**問題**: 実行時エラーや保存失敗が console 止まりで、誰にも見えない。

**手法**: `window.onerror` / `unhandledrejection` を捕捉して userData 配下の
ローカルログへ記録（**外部送信は一切しない**）。保存失敗（`Store.lastSaveOk` は既存）を
トースト通知に昇格。

## フェーズ2: 録画エンジン v2 — 「画面を撮る」から「操作を理解する」へ（R1〜R4）

**SaaS対抗の核。** 現在の録画は「座標を取ってスクショを撮る」だけで、その後の
文章書き・画像整形がすべて手作業として残る。Scribe / Tango との差はここで生まれている。
録画を**操作の意味を記録するエンジン**に作り替える。

### 2-R0. UIA 検証スパイク — 規模 S（1〜2日）【✅ 実施済み 2026-07-13・判定 go】

> **結果**: `docs/spike-2-R0-results.md` 参照。Office系 96〜100%（Excel はセル番地
> まで取得可）・Windows標準 100%・ブラウザ 75% で基準を満たし
> **R2 は計画どおり進める**（4カテゴリの計測完了）。重要な発見: RemoteApp（リモートデスクトップ）
> 経由の業務システムは要素がウィンドウ単位までしか見えず、**フォールバック文生成が
> 主系統になる**（前提10 の実証）。koffi＋UIA の本番経路は実機で成立
> （解決 5〜60ms、非同期解決で隠せる）。計測ツールは `tools/uia-spike/`。

**目的**: フェーズ2の核（UIA による要素解決）がどれだけ成立するかを、本実装の前に実測で
確かめる。結果が R2/R3 のフォールバック設計と期待値、ひいては計画全体の見通しを決める。

**実施体制**（開発環境は Linux コンテナのため、Windows 実機の作業はユーザーが担う）:

1. Claude が計測スクリプトを作成する — koffi（FFI）＋ UIA `ElementFromPoint` で、
   クリックごとに「要素名 / 種類(ControlType) / 矩形 / 親ウィンドウタイトル / アプリ名 /
   取得可否」を JSON に記録する計測ツール。
2. ユーザーが Windows 実機で実行し、下記4カテゴリの対象アプリ（計10種目安）を
   ふだんの操作で数分ずつクリックする。
3. 結果 JSON を開発セッションに貼り戻し、カテゴリ別の要素名取得率を集計する。

**対象アプリ（2026-07-11 確定: 4カテゴリすべて）**:
①Office系（Excel / Word / Outlook 等）
②ブラウザ内の業務システム（Chrome / Edge 上の Web アプリ）
③独自の業務デスクトップアプリ
④Windows標準（エクスプローラー / 設定 等）。

**go/no-go 判断基準（初期値。実測を見て調整してよい）**:

| 実測結果 | 判断 |
| --- | --- |
| Office・Windows標準で要素名取得率 **70%以上** | R2 は計画どおり進める |
| ブラウザ内アプリの取得率が **50%未満** | UIA ブリッジ（Chromium のアクセシビリティ有効化条件）や代替手段を追加調査してから R2 を設計する |
| 独自業務アプリの取得率が低い（想定内） | フォールバック文生成（ウィンドウタイトル＋座標切り出し）を「例外処理」ではなく**主系統の一つ**として R2/R3 を設計する |
| 全カテゴリで壊滅的（目安: 平均3割未満） | テンプレート文生成の期待値を下げ、R3 は座標中心切り出し主体へ倒す。到達レベルの定義自体をユーザーと再協議する |

### 2-R1. 録画セッション形式（画像＋メタデータ併記） — 規模 S【✅ 実施済み 2026-07-13】

> **結果**: `session.js`（新規）で「録画1回 = 1セッションフォルダ（ピクチャ配下は維持）」＋
> `001.png`/`001.json` 併記＋`session.json` を実装。サイドカーには座標・時刻・ボタン・
> 連続クリック数・ディスプレイ・マーカー情報と、R2 で埋める `uia` の雛形を記録。
> マーカー焼き込みは既存トグルのまま変更なし（素の画像の複製は持たない）。
> 確定仕様は `docs/spec-2-R1-session-format.md`、テストは `test/session.test.js`。

**問題**: 撮影情報（クリック座標・時刻・対象ディスプレイ）を取得しているのに
PNG 保存後に**捨てている**。後段の全機能がこの情報を必要とする。

**手法**: 1クリックごとに JSON サイドカーを併記する。

```
録画セッション/<リスト名_日時>/
  001.png
  001.json   ← 座標、UI要素名、種類、要素矩形、ウィンドウタイトル、アプリ名、時刻
  002.png ...
```

- `persistShot`（main.js）で PNG と同時に JSON を書くだけ。R2 以前は UIA 項目が
  空のまま座標・時刻・ウィンドウ情報だけでも先に保存を始める。
- **効果**: 取り込み処理を後から何度でもやり直せる（取り込みロジックを改良したら
  過去の録画も恩恵を受ける）／デバッグ容易／R2〜R4・フェーズ4が全部この上に載る。

### 2-R2. UI要素解決＋テンプレート文生成（本丸・コア） — 規模 L【✅ 実装済み 2026-07-13・実機検証待ち】

> **結果**: `uia-host.js`（utilityProcess・koffi＋UIA。2-R0 計測ツールの本番化）＋
> `uia.js`（親側ラッパ。タイムアウト2秒・クラッシュ隔離・非対応環境は null）＋
> `steptext.js`（純関数のテンプレート文生成。Excel セル特化文・コンテナ系は
> フォールバック）を実装。mousedown で撮影と並行に非同期解決し、サイドカー v2 に
> `uia`（実データ）と `text`（生成文）を記録する。確定仕様は
> `docs/spec-2-R2-uia-steptext.md`、テストは `test/steptext.test.js`（全64テスト成功）。
> **残タスク**: ユーザーの Windows 実機での動作確認（Chrome ページ内解決の再計測含む）。

**問題**: 手順テキストを全部ユーザーが書いている。SaaS は「設定画面で『保存』を
クリック」と**自動生成**しており、ここが最大の体験差。

**手法**

- **UI Automation (UIA)**: Windows 標準 API の `ElementFromPoint` で、クリック座標から
  UI 要素を取得する——要素名（「保存」）、種類（button/tab/edit/checkbox/menuitem）、
  **要素の矩形**、親ウィンドウのタイトル、アプリ名。
- 呼び出し手段は **koffi（FFI・コンパイラ不要）**から始める。性能や安定性で不足したら
  C++ N-API アドオンへ昇格。mousedown 時に撮影と**並行して非同期解決**すれば
  レイテンシは隠せる（既存の mousedown→mouseup 二段設計と相性が良い）。
- **テンプレート文法による決定的生成**（AI不要・オフライン・幻覚なし）:

  | 取得した情報 | 生成される文 |
  | --- | --- |
  | button「保存」 | 「**保存**」ボタンをクリック |
  | tab「ホーム」 | 「**ホーム**」タブを選択 |
  | edit「ファイル名」＋入力検出 | 「**ファイル名**」に入力 |
  | checkbox「同意する」 | 「**同意する**」にチェック |
  | menuitem「印刷」 | メニューから「**印刷**」を選択 |
  | 要素名なし（フォールバック） | ウィンドウ「◯◯」内の図の位置をクリック |

- **フォールバック設計と期待値は 2-R0 の検証スパイク結果に従う**。一部アプリ
  （ゲーム・独自描画・一部の独自業務アプリ）では取れないことを前提に、常に
  ウィンドウタイトル＋座標へフォールバックできる設計にする（前提10）。

**効果**: 「録画したら文章付きの手順書が出来ている」の核が成立。SaaS の核心機能と
オフラインで並ぶ。

### 2-R2b. 操作種類の拡張（キーボード・ドラッグ・アプリ切替） — 規模 M【✅ 実装済み 2026-07-13・実機検証待ち】

> **結果**: ①〜⑤すべて実装（ユーザー確定: 全部・入力確定時に撮影・入力内容は
> 一切記録しない・アプリ切替はサイドカー記録のみ）。`keys.js`（新規・キー分類の
> 純関数）＋ `steptext.js` の文生成拡張＋サイドカー **v4**（kind/keys/drag/appChange）
> ＋ `uia` のフォーカス要素解決＋ガジェットの「ドラッグを記録する」トグル（既定OFF）
> ＋ウィザードの追従（ドラッグは 始点/終点/両方）。ダブルクリックはデバウンス集約後の

## フェーズ3.5: 配布と信頼 — 「配れる品質」を製品の一部にする

機能と同じくらい、**入手のしやすさ・疑われないこと・更新の透明さ**が SaaS との比較点になる
（SaaS はインストール不要・更新は透明。こちらは無署名インストーラ＋グローバル入力フックで
その真逆から始まる）。品質基準は配布先で変えない方針（前提7）に基づき独立フェーズとして
計画に含める。中間ゴール（R1〜R4）達成の前後で着手する。

### 3.5-A. コード署名（調査 → ユーザー判断） — 規模 S（調査まで）【✅ 調査済み 2026-07-13・✅ 判断済み 2026-07-14: 購入見送り】

> **ユーザー決定（2026-07-14）**: 本開発は**一切お金をかけない**方針のため、証明書購入は
> 見送り（無署名配布を継続）。無償の緩和策（SmartScreen 手順と挙動の明記・SHA-256
> チェックサム添付・ソース公開の透明性）のみ実施。Azure Artifact Signing の日本開放時に
> 再検討する。詳細は `docs/spec-3.5-A-code-signing.md` 冒頭の決定記録を参照。

> **結果**: `docs/spec-3.5-A-code-signing.md` 参照。要点: ①EV の SmartScreen 即時信頼は
> 2024 年に廃止され OV と同等（EV は過剰投資）②最安の Azure Trusted Signing
>（現 Artifact Signing）は日本が対象外 ③現実解は法人なら国内 OV（GMO/Sectigo、
> 5.5〜6万円/年）か SSL.com OV＋eSigner（CI 完全自動化）④どれでも警告は即消えせず
> 同一証明書での継続署名でレピュテーションを積むのが本質 ⑤証明書の CN を変えると
> 自動更新が壊れるため、CN 固定 → 3.5-B 着手の順序を守る。**購入・契約の判断はユーザー。**

**問題**: 無署名の NSIS インストーラ＋グローバル入力フック（uiohook-napi）は、
SmartScreen 警告と AV 誤検知に正面からぶつかる。README での挙動説明だけでは
初見のユーザーはインストールまで辿り着けない。

**手法**: OV/EV コード署名証明書・Azure Trusted Signing 等の選択肢を、費用・取得要件
（法人/個人）・SmartScreen への実効性込みで1ドキュメントに調査してまとめる。
**購入・契約の判断はユーザーが行う**（外向き支出のため独断しない。運用ルール領域4）。

### 3.5-B. 自動更新 — 規模 M

**手法**: **electron-updater ＋ GitHub Releases**。起動時に更新をチェックし、
ユーザー確認の上で適用する（サイレント強制更新はしない）。実運用ではコード署名
（3.5-A）が前提になる構成が望ましいため、順序は 3.5-A の後。

### 3.5-C. リリース工程とバージョニング — 規模 S

**手法**: フェーズ完了とバージョン番号を対応付ける（例: フェーズ1完了 = v1.1、
中間ゴール（R1〜R4）= v2.0。現状は 1.0.0 のまま動いている）。CHANGELOG を整備し、
GitHub Releases にビルド済みインストーラを添付する運用を確立する
（既存の `build-windows.yml` を土台に）。

## フェーズ4: 発展モジュール — 「AI込みで完全オフライン」（R7〜R8）

### 4-R7. ローカルLLMによる文章仕上げ（任意モジュール） — 規模 L

**問題**: テンプレート生成文（2-R2）は正確だが、「クリック」の連続で文章としては
機械的。SaaS は生成AIでタイトル・要約・自然な文章まで作る。

**手法**

- **node-llama-cpp ＋ 小型日本語対応モデル（数GB）**を**任意ダウンロードの
  オプションモジュール**として統合。既定は無効（本体の軽さを守る）。
- 用途は絞る: ①テンプレート文の自然文化・丁寧語化、②手順全体からのタイトル・概要・
  注意書きの下書き生成、③「新人向けに言葉を柔らかく」等の一括リライト。
- **推論も完全ローカル。データは1バイトも外に出ない。**「AI仕上げまで含めて
  オフライン」はクラウドSaaSに原理的に真似できない売り文句になる。

### 4-R8. 機密情報マスキング支援 — 規模 M〜L

**問題**: 機密のぼかしが手動モザイクのみ。SaaS は自動でぼかす。

**手法**: **tesseract.js（ローカルOCR・日本語対応）**で画像内テキストを検出し、
メールアドレス・電話番号等のパターン一致領域を「モザイク候補」として提示、
ワンクリック適用。処理が重いためバックグラウンド実行＋候補提示型のUXにする。

## フェーズ5: 使い込みに耐える — スケールと操作性

リストが増え、日常的に使われる段階で効く改善。**フェーズ2〜4の合間に S 規模のものから
差し込んでよい**（依存が薄いため）。

- **5-1. 全文検索（S）**: ホームに検索ボックス。タイトル・項目・メモを横断する
  インクリメンタル検索。データは全件メモリ上にあるため**索引不要の線形走査で十分**
  （数百リスト規模まで）。ヒットのハイライトとエディタ内ジャンプまでやると完成度が高い。
- **5-2. ホーム整理: タグ＋アーカイブ（M）**: model に `tags: string[]` と
  `archived: bool` を追加（旧データは `Store` 初期化時のデフォルト補完——`settings`
  補完と同じ既存パターン）。フォルダ階層よりタグが model 変更が小さく D&D 移動も
  不要なので、先にタグで様子を見る。
- **5-3. キーボード操作とアクセシビリティ（S〜M）**: `Ctrl+F`（検索）、`Enter`
  （次の項目を追加・`addBlankItem` 流用）、`Ctrl+P`（印刷）、`Esc`（閉じる・一部実装済み）。
  モーダルのフォーカストラップ、aria ラベル、両テーマのコントラスト検証（WCAG AA 目安）。
- **5-4. 描画性能（M・計測してから）**: `render()` は全再描画方式。**先に計測**
  （項目数 100/500/1000 で操作）し、体感遅延が出てから部分再描画化する。
  **問題が出ていない段階での最適化はしない。**

## 追わないと決めた領域（Tier C）

以下は**クラウドであることが本質**の機能であり、オフライン・ローカル完結の製品が
追うと戦略が壊れる。「やらない」と明記しておく。

- 閲覧分析（誰がいつ見たか）
- 承認ワークフロー
- 多言語自動翻訳（クラウド翻訳API前提のもの）
- リアルタイム共同編集

## 技術リスク（継続的に把握しておくこと）

| リスク | 内容と対応方針 |
| --- | --- |
| UIA の当たり外れ | アプリによって要素名が取れない（ゲーム・独自描画・一部の独自業務アプリ）。**検証スパイク（2-R0。計画全体の最初）で実測**し、ウィンドウタイトルへのフォールバックを常に用意する。 |
| koffi（FFI）の安定性 | クラッシュはメインプロセスを巻き込む。UIA 呼び出しは try/catch と時間制限で包み、失敗時は「要素情報なし」で続行。不足なら C++ N-API アドオンへ昇格。 |
| ローカルLLM のサイズと品質 | 数GBのモデル配布・小型モデルの日本語品質は要検証。任意モジュール（既定無効）とし、テンプレート生成（R2）だけで実用になる設計を崩さない。 |
| `html-to-docx` のメンテ停滞 | Word 出力の土台。3-C の調査で移行パスを確保しておく。 |
| Electron の追随更新 | 現在 31 系。セキュリティ修正が頻繁なため定期的なメジャー追随を運用に組み込む（テスト整備後はコストが下がる）。 |
| 単一 HTML の行数増加 | レンダラーは方針として維持。既存のセクション構成（model/storage/io/render/…）と関数粒度の規律で対処。**メインプロセス側は分割を許容**（前提2）。 |
| `uiohook-napi` の AV 誤検知 | グローバル入力フックは疑われやすい。挙動（クリック座標のみ・外部送信なし）を README/ガイドに明記した上で、実効対策はコード署名（3.5-A）で行う。 |
