# CheckListMaker 品質向上ロードマップ

アプリのクオリティを商用水準へ引き上げるための開発計画。**機能の思いつき順ではなく、
「データを失わない → 中核体験を完成させる → 使い込みに耐える」の順**で積み上げる。

各項目は「**問題 → 手法 → 効果 → 規模感**」の形式で書く。規模感の目安:
**S**=半日〜1日 / **M**=数日 / **L**=1〜2週間（検証込み）。

## 前提（確定済みの技術判断）

本ロードマップは以下の決定を前提とする（2026-07-11 確定）。

1. **単一HTML構成を維持する。** ビルド工程（バンドラ）は導入しない。
   `index.html` 1ファイル・依存ゼロ・`file://` 動作という構成上の利点を守る。
2. **保存の移行先は「JSON＋画像個別ファイル」。** SQLite 等のネイティブDBは使わない
   （ネイティブ依存を増やさない。`uiohook-napi` の再ビルド運用だけで抑える）。
3. **ブラウザで `index.html` 単体を開いても動く特性を現状同等に維持する。**
   Electron 起動時＝ファイル保存、ブラウザ単体＝従来どおり localStorage、の2系統。
4. **実施記録（点検ログ）は当面作らない。** まず手順書作成体験の品質に集中する
   （フェーズ3の「将来枠」に記載のみ）。

## フェーズ1: 信頼性の土台 — データを失わない・壊れない

機能を増やす前に、いま抱えている構造的な弱点を塞ぐ。**このフェーズを最初にやる理由は、
後回しにするほど移行の難易度と事故リスクが上がるため**（データ形式の変更は、データが
増えるほど・機能が積み重なるほど難しくなる）。

### 1-1. 保存基盤の移行（最優先） — 規模 L

**問題**

- 全データ（画像含む）を localStorage（実質約 5MB）に保存している（`index.html` の
  `LocalStorageAdapter`）。画像は長辺 1600px・JPEG 品質 0.85 の dataURL（1枚 150〜400KB
  程度）で state に直接埋め込まれるため、**写真を数十枚貼った時点で保存が失敗し始める**。
  「録画してスクショを貼る」が売りのアプリとして設計矛盾。
- Undo が「画像込みの全 state」を最大 50 世代フルクローンして保持する（`Store.commit`）。
  画像が増えるとメモリが数百MB規模に膨らみ得る。

**手法**

- **Electron 側（メインプロセス）**
  - `preload.js` に contextBridge で `window.storageAPI` を追加する。既存の
    `recorderAPI` と同じパターン（`contextIsolation: true` のまま安全に公開）。
  - `main.js` に IPC ハンドラを追加:
    `storage:load` / `storage:save` / `image:save` / `image:get` / `image:delete`。
  - 保存先はユーザーデータフォルダ:
    - 本体: `app.getPath('userData')/data/checklists.json`
    - 画像: `app.getPath('userData')/data/images/<uuid>.jpg`（1枚1ファイル）
  - 書き込みは**アトミック方式**（一時ファイルに書いて `fs.renameSync`）＋
    直前世代を `checklists.json.bak` として保持。書き込み途中のクラッシュでも
    直前状態に戻れる。
- **レンダラー側（index.html）**
  - 既存の `StorageAdapter` 抽象（`LocalStorageAdapter` / `MemoryAdapter` が実装済み）に
    **`FileAdapter` を追加**する。起動時に `window.storageAPI` の有無で自動選択:
    - Electron → `FileAdapter`（ファイル保存）
    - ブラウザ単体 → 従来どおり `LocalStorageAdapter`（**無改修で互換維持**）
    - 書き出した単体HTML → 従来どおり `MemoryAdapter`
  - 画像の持ち方を変更: `item.images` の dataURL を **`img:<uuid>` の参照**にする
    （Electron モードのみ）。表示時に `image:get` で解決し、Blob URL でキャッシュ。
    ブラウザモードは従来どおり dataURL 直持ち（5MB の範囲で動く現状仕様のまま）。
  - **入出力の互換を守る**: JSON エクスポート・単体HTML書き出し（`buildStandaloneHtml`）の
    際は参照を dataURL に戻して埋め込む。逆にインポート時は dataURL を画像ファイルへ
    展開して参照化する。→ **既存のバックアップ・共有ファイルはそのまま使い続けられる**。
- **マイグレーション**
  - Electron 初回起動時に localStorage の既存データ（`checklistmaker.v1`）を検出したら、
    ファイルへ一括変換（画像は個別ファイルへ展開）し、完了マーカーを localStorage に書く。
    変換前の localStorage は消さずに残す（切り戻し可能にする）。
- **Undo の軽量化**
  - 画像が参照（短い文字列）になるため、`Store` の 50 世代クローンは自然に軽くなる。
    追加の対応は不要になる見込みだが、`imageEdits`（注釈の元データ）も同様に
    参照化することを忘れない。

**効果**: 容量上限の実質撤廃／保存失敗・データ消失リスクの解消／Undo のメモリ膨張解消／
1-1 完了により 2-3（画像画質）と将来の履歴系機能の土台ができる。

### 1-2. 自動テストと CI — 規模 M

**問題**: 5,130 行の単一 HTML に自動テストが 0 本。今後の改修（特に 1-1 のような
危険な変更）を目視確認だけで進めるのはリスクが高すぎる。

**手法**（単一HTML維持の制約下でのやり方）

- **テストハーネス方式**: `test/harness.js` が `index.html` を読み込み、`<script>` 部分を
  抽出して jsdom 環境で評価する。アプリ側には
  `if (typeof window.__test__ !== 'undefined') { window.__test__.expose({ ... }) }` の
  ようなフックを1箇所だけ足し、テスト時のみ内部関数（model / io の純関数）を公開する
  （本番動作には一切影響しない）。
- テストランナーは **Node 標準の `node:test`**＋jsdom。依存を最小に保つ
  （Vitest 等のフレームワーク導入はビルド工程を持たない方針と整合しないため見送り）。
- **対象の優先順位**:
  1. 1-1 のマイグレーション処理（最重要。既存ユーザーデータを壊さない保証）
  2. model: 項目/セクションの追加・移動・削除、進捗計算、一括リセット
  3. io: JSON入出力、単体HTML書き出し/取り込み、URL共有のエンコード/デコード
- **CI**: `.github/workflows/` にテスト用 workflow を追加し、push / PR ごとに
  Linux 上で実行（Windows ビルドの workflow とは分離。テストは数十秒で回る想定）。

**効果**: リグレッションの自動検知。1-1 を安心して実施できる前提条件。

### 1-3. エラーの可視化 — 規模 S

**問題**: 実行時エラーや保存失敗が console.warn 止まりで、ユーザーにも開発者にも
見えない。「保存できていなかったことに後で気づく」が最悪のパターン。

**手法**

- `window.onerror` / `unhandledrejection` を捕捉し、userData 配下のローカルログ
  （`logs/error.log`、ローテーションあり）へ記録。**外部送信は一切しない**
  （オフライン完結の方針どおり）。
- 保存失敗（`Store.lastSaveOk` は既にある）を画面上のトースト通知に昇格させ、
  その場で気づけるようにする。

**効果**: 不具合の早期発見と、ユーザーからの障害報告時に調査可能になる。

## フェーズ2: 中核体験の完成 — 録画→手順書が一気通貫

このアプリの独自価値は「クリック連動録画 → 注釈 → 印刷/Word/PDF」。現状はこの鎖の
**録画と手順書の間が手作業で切れている**。ここを繋ぐ。

### 2-1. 録画→手順の一括取り込み（体験上の最重要機能） — 規模 M

**問題**: 録画は「スクショがフォルダに溜まる」まで。その後ユーザーは手順を1つずつ
作り、画像を1枚ずつ手で貼る必要がある。競合ツール（Scribe / Tango 等）の支持理由は
「操作し終えたら手順書がもう出来ている」体験であり、ここが最大の差。

**手法**

- `main.js` の録画セッション中、保存したファイルパスを配列に蓄積する
  （`persistShot` に1行追加する程度）。
- `rec:stop` の戻り値（または `rec:state` イベント）でファイル一覧をレンダラーへ返す。
- レンダラーは録画停止時に「撮影した◯枚を手順として取り込みますか？」ダイアログを表示。
  承諾したら **1クリック＝1ステップ**として、`createItem`＋画像添付を自動生成する。
  画像の読み込み・圧縮・添付は既存の `compressImage` / `addItemImage` をそのまま流用。
- 取り込み先は「開いているチェックリストの新しいセクション（録画日時をタイトルに）」を
  既定とし、取り込み後にユーザーが並べ替え・文言修正する流れ。
- ブラウザモードでは録画ボタン自体が無効なので、分岐の追加は不要。

**効果**: 手順書作成の所要時間が劇的に短縮され、「録画したら手順書の骨格ができている」
というこのアプリの完成形に到達する。実装規模に対する価値の増分が全項目中で最大。

### 2-2. Excel / CSV 出力 — 規模 M

**問題**: Word 出力は表紙・目次・改訂履歴まで作り込まれている一方、日本の現場の
点検表・チェック記録は圧倒的に Excel 文化。「Excelで出せないか」は業務利用で
必ず出る要望。

**手法**

- **exceljs**（純JS・ネイティブ依存なし）で .xlsx を生成する。列構成は
  「No / 項目 / チェック / 所要時間 / メモ」を基本に、セクションを見出し行で表現。
  表紙情報（タイトル・版数・作成者）はシート先頭またはプロパティに反映。
- CSV は **BOM 付き UTF-8** で出力（Excel でダブルクリックしても文字化けしない）。
- Electron 側の保存 IPC は既存の `docx:save` と同型（`dialog.showSaveDialog`→書き込み）。
  ブラウザモードでは既存 `downloadHtml` と同じ Blob ダウンロード方式で対応可能。

**効果**: 業務現場での出力ニーズの本命に応える。Word 出力と合わせ主要形式を網羅。

### 2-3. 画像画質の向上 — 規模 S〜M（1-1 完了が前提）

**問題**: localStorage の容量制約のため、画像は取り込み時に長辺 1600px・JPEG 0.85 へ
不可逆圧縮している。印刷や Word 出力で文字の細かいスクショが潰れることがある。

**手法**: 1-1 で画像がファイル保存になったら、**原寸の元画像を保存**し、画面表示用には
縮小サムネイルを別途生成する2段構成へ変更。印刷/Word/PDF 出力時は原寸を使う。
`IMG_MAX_EDGE` / `IMG_QUALITY` は「表示用」の設定として残す。

**効果**: 手順書の実用品質（読めるスクショ）を担保。

### 2-4. Word 出力の依存リスク対応（調査タスク） — 規模 S（調査のみ）

**問題**: Word 出力が載っている `html-to-docx` はメンテナンスが停滞気味で、
不具合修正が見込みにくい。主要機能が塩漬けライブラリに依存している状態。

**手法**: 活発にメンテされている **`docx`** パッケージへの移行可能性を調査する
（現在の後処理 `docx-postprocess.js` で行っている様式調整が `docx` なら生成時に
直接表現できる見込み）。**すぐ書き換えるのではなく**、現状の出力サンプル一式を
基準（ゴールデンファイル）として保存した上で、移行コストと得失を1ドキュメントに
まとめてから判断する。

**効果**: 将来 Word 出力が壊れた際に詰まない。移行するなら後処理ハックの解消も狙える。

## フェーズ3: 使い込みに耐える — スケールと操作性

リストが増え、日常的に使われるようになった段階で効く改善。

### 3-1. 全文検索 — 規模 S

**問題**: 検索機能がない。手順書が 20〜30 件を超えると「あの手順どこ？」が日常になる。

**手法**: ホーム画面に検索ボックスを追加し、タイトル・項目テキスト・メモを横断する
インクリメンタル検索を実装。データは全件メモリ上にあるため、**索引不要の線形走査で
十分**（数百リスト規模まで問題ない）。ヒット箇所のハイライトと、エディタ内ジャンプまで
やると完成度が高い。

### 3-2. ホーム一覧の整理（フォルダ/タグ＋アーカイブ） — 規模 M

**問題**: ホームはカードがフラットに並ぶだけで、増えると破綻する。

**手法**: model にフィールドを追加（例: `checklist.tags: string[]` と
`checklist.archived: bool`）。旧データは `Store` 初期化時のデフォルト値補完で対応
（`settings` 補完と同じ既存パターン）。ホームにタグフィルタとアーカイブ切替を追加。
フォルダ階層よりタグ方式のほうが model 変更が小さく、D&D 移動も不要なので先にタグで様子を見る。

### 3-3. キーボード操作とアクセシビリティ — 規模 S〜M

**問題**: マウス前提の操作体系。業務で毎日使うツールはキーボード完結度が効率を左右する。
また aria 属性やフォーカス管理が未整備。

**手法**: まず高頻度操作から: `Ctrl+F`（検索）、`Enter`（次の項目を追加・既存の
`addBlankItem` を流用）、`Ctrl+P`（印刷プレビュー）、`Esc`（ダイアログを閉じる・一部実装済み）。
モーダルのフォーカストラップ、チェックボックスとボタンへの aria ラベル付与、
ライト/ダーク両テーマでのコントラスト検証（WCAG AA 目安）を段階的に。

### 3-4. 描画性能 — 規模 M（計測してから着手）

**問題（潜在）**: `render()` は全画面を毎回作り直す方式。項目数が数百を超えるリストや
画像の多いリストで、1操作ごとの再描画が重くなる可能性がある。

**手法**: **先に計測**（Performance パネルで項目数 100/500/1000 のリストを操作）し、
体感遅延が出るラインを確認してから着手する。対策の候補は、操作のあったセクションのみの
部分再描画、画像の遅延読み込み（`loading="lazy"` は導入済みのため Blob URL 解決の遅延化が主）。
**問題が出ていない段階での最適化はしない**。

### 3-5. 将来枠（当面着手しない。順位付けはユーザー判断で後日）

- **実施記録（点検ログ）**: 一括リセット時に実施結果（日時・チェック状態・メモ）を
  アーカイブし、履歴一覧・出力を可能にする。監査・報告用途の本命機能だが、
  当面不要と判断済み（2026-07-11）。1-1 のファイル保存化が実現すれば容量面の障害は無い。
- **スケジュール／リマインド**: 「毎週月曜に自動リセット」「点検日に通知」。
- **テンプレートギャラリー**: 業種別ひな形（日次設備点検・新人受け入れ等）の同梱。
  機能開発というよりコンテンツ制作。
- **出力様式のカスタマイズ**: ロゴ・ヘッダー/フッター・Excel 点検表レイアウトの調整。

## 実施順序のまとめ

```
フェーズ1（土台）   : 1-1 保存移行 → 1-2 テスト+CI（1-1と並行可） → 1-3 エラー可視化
フェーズ2（中核）   : 2-1 録画→手順取り込み → 2-2 Excel/CSV → 2-3 画像画質 → 2-4 docx調査
フェーズ3（操作性） : 3-1 検索 → 3-2 タグ/アーカイブ → 3-3 キーボード/a11y → 3-4 性能(計測後)
```

順序の論理: **データを失わない土台（1）がすべての前提。次に独自価値を完成させ（2）、
最後に日常利用の摩擦を削る（3）。** 各フェーズ末でひとつ前のフェーズの回帰テストを回す。
