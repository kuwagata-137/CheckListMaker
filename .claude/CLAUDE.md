# プロジェクト運用ルール

## 開発フロー（PR 経由）

- **変更は原則すべて PR 経由で行う。**「作業ブランチを切る → コミット → PR を作成 → `main` にマージ」という流れを守る。
- **`main` へ直接 push しない。** ユーザーの明示的な許可がない限り、勝手に `main` へ直接コミット／push してはいけない（Git を使う意味がなくなるため）。
- **PR の作成・マージは、ユーザーが明示的に指示するまで行わない。** 作業ブランチへのコミット・push までは進めてよいが、**PR を立てる／マージするのは必ずユーザーの指示を待つ**。勝手に PR 作成やマージをしない。

## ブランチ統合ポリシー

- 作業ブランチを増やして散らかさない。**機能・修正は基本的にすべて `main` にマージして統合する。**
  - ブランチが分散していると「main に何の機能があるか」が誤解されるため。
- 機能を別ブランチ（や別ブランチ宛の PR）に置いたまま放置しない。完成したら `main` に取り込む。
- **統合（マージ）が完了したら、不要になったブランチ（ローカル／リモートとも）は削除する。**
  - 例外: `main` は削除しない。作業中で未統合のブランチは残してよい。
