# プロジェクト運用ルール

このプロジェクトでの開発方針を定める。下記は **MECE（重複なく・漏れなく）** で整理した
8つの領域からなる。各ルールはいずれか1領域にのみ属する。Claude（および作業者）は
本方針に従うこと。これらは既定の挙動より優先される。

---

## 1. 着手前 — 方向性と仕様の確定

実装に入る前に、まず方向と仕様を固める。コードを書き始めるのは最後。

- **ロードマップ（方向性）を先に再確認する。** 何を作るのか、今やるべきことは何かを
  共有・合意してから動く。いきなり実装に飛び込まない。
- **曖昧な仕様を残したまま実装しない。** 仕様書・要件に曖昧点や矛盾があれば、実装前に
  すべて洗い出し、ユーザーの判断を仰いで確定させる。
- 「この時点で、ほかに詰めるべき仕様はないか？」を着手前に必ず自問・確認する。
- 確定した決定事項は、後続が迷わないよう仕様ドキュメント等に書き残してから実装に進む。

## 2. ローカル作業 — ブランチとコミットの進め方

ローカルかつ可逆な作業は、ユーザーの逐次指示を待たずに進めてよい領域。

- **作業ブランチを切ってそこで作業する。** `main` 上で直接作業しない。
- **既存の作業ブランチがあれば再利用する。** 用もなく新しいブランチを増やさない
  （散らかり防止＋リモートブランチは消せないため。→ 領域6・領域8参照）。
- コミットは小さく、メッセージは何をなぜ変えたかが分かるように書く。
- 作業ブランチへの **commit・push までは進めてよい**（push 自体は外向きだが、作業ブランチの
  更新は可逆で安全なため許可。ただし `main` への push は領域3で禁止）。

## 3. 統合ゲート — PR・マージ・main 反映（ユーザー承認が必須）

「main に何が入るか」を決める操作は、すべてユーザーがゲートを握る。Claude は独断で行わない。

- **`main` へ直接コミット／push しない。** ユーザーの明示的な許可がない限り禁止
  （Git を使う意味がなくなるため）。
- **PR の作成は、ユーザーが明示的に指示するまで行わない。** 勝手に PR を立てない。
- **PR のマージは、ユーザーが明示的に指示するまで行わない。** 勝手にマージしない。
  自動マージ（auto-merge）の有効化も同様に指示待ち。
- 「作業ブランチへの push までは Claude、PR を立てる／マージするのはユーザー」を境界とする。
- これらは settings.json のフックでも機械的に強制している（領域8参照）。
  **`main` 直 push は遮断（deny）／PR 作成・マージは要確認（ask）**。フックに頼らず、まず本ルールを守ること。

## 4. その他の外向き・不可逆操作 — 独断禁止

領域3（PR・マージ）以外で、外部に影響する／戻しにくい操作も独断で行わない。

- 履歴の書き換え（force push、`rebase` 後の強制更新、`reset --hard` での共有履歴改変）は
  事前確認する。
- 外部サービスへの送信・公開（パッケージ公開、デプロイ、外部 API への投稿・コメント等）は
  事前確認する。一度出したものはキャッシュ／インデックスされ得る。
- 既存ファイル・データの削除や上書きは、対象を確認し、説明と矛盾する内容や自分が作って
  いないものなら、進めずに報告する。
- 迷ったら「やってよいか」を先に確認する。承認は文脈ごと。一度の許可を別の操作へ広げない。

## 5. ブランチのライフサイクル — 整理と統合

ブランチを散らかさず、完成したら main に集約する。

- 機能・修正は基本的にすべて `main` にマージして統合する。別ブランチに置いたまま放置しない
  （どの機能が main にあるかを誤解させないため）。
- **統合（マージ）が完了したら、不要になったブランチは削除する。**
  - ローカルブランチは通常どおり削除してよい。
  - リモートブランチの削除はこの環境では不可（領域6参照）。ユーザーに依頼する。
  - 例外: `main` は削除しない。作業中で未統合のブランチは残してよい。

## 6. 実行環境の制約 — リモートブランチ削除は不可

この実行環境固有の制約。知らないと無駄なゴミブランチを生む。

- **この環境ではリモートブランチを削除できない。** `git push --delete` は git プロキシが
  **HTTP 403** で拒否し、GitHub MCP にも削除用ツールが無い（`create_branch` はあるが
  `delete_branch` 系は無い）。**認証や `gh` 未インストールが原因ではない**——作成・更新・
  コミット・push は正常にできる。
- したがって、**切り分け・テスト目的でも、使い捨てのリモートブランチを作らない**
  （消せずに残るため。代わりに既存ブランチを再利用する。→ 領域2）。
- リモートブランチの削除が必要なときは、**ユーザーに GitHub Web UI での削除を依頼する**
  （`https://github.com/<owner>/<repo>/branches`）。

## 7. コミュニケーションと報告

進め方と結果の伝え方。

- **結果は正直に報告する。** テストが失敗したら出力とともにそう言う。スキップした手順は
  スキップしたと言う。完了・検証済みなら、はっきりそう言い切る。
- **ミスは言い訳せず認める。** 取り繕わない。
- 食い違い・予期せぬ結果・判断の分岐は、**なぜそうなったかを筋道立てて説明**する。
- 判断がユーザーに属するとき（方向・仕様・外向き操作）は、勝手に決めず確認する。

## 8. 方針の維持と伝播

決めたことを毎回ブレさせず、別の開発にも伝える。

- 合意した運用方針は本ファイル（`.claude/CLAUDE.md`）に記録し、リポジトリと共に伝播させる。
- 同じ方針をユーザー全体メモリ（`~/.claude/CLAUDE.md`）にも置き、ローカルの全プロジェクトへ
  効かせる（ただし Web/リモート環境ではコンテナ再生成で消えるため、リポジトリ内が確実）。
- 機械的強制として、`.claude/settings.json` の **PreToolUse フック**で
  **`main` 直 push を遮断（deny）**、**PR 作成・マージを要確認（ask）** に倒す（領域3の保険）。
- 新しいリポジトリへは、本方針を運ぶスキル **`dev-policy`** を使って、`.claude/CLAUDE.md`・
  `.claude/settings.json` をひな形として展開する。
