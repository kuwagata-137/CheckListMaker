# 2-R0 UIA検証スパイク — 計測ツール

クリックのたびに Windows UI Automation（UIA）でクリック先の UI 要素を解決し、
**「要素名（『保存』など）がどれだけ取れるか」をアプリ別に実測する**ツールです。
録画エンジン v2（ロードマップ 2-R2）の設計と期待値を、実装前に実測で決めるために使います。

- 記録するもの: 要素名 / 種類(ControlType) / 要素の矩形 / ウィンドウタイトル / アプリ名 /
  取得可否 / 解決の所要時間（クリック座標・時刻含む）
- **外部送信は一切ありません。** 結果はこのフォルダ内のファイルに書かれるだけです。

## 準備（Windows 実機・1回だけ）

1. [Node.js LTS（64bit）](https://nodejs.org/ja/) をインストール
2. このリポジトリを取得し、コマンドプロンプト（または PowerShell）でこのフォルダへ移動

   ```bat
   cd <リポジトリの場所>\tools\uia-spike
   npm install
   ```

## 実行手順

```bat
npm run selftest   … 動作確認（カーソル位置の要素を1回だけ解決して表示）
npm run measure    … 計測開始
```

計測が始まったら、**ふだんの操作どおり**に対象アプリを数分ずつクリックしてください。
1クリックごとにコンソールへ「アプリ名 | 要素種類 | 要素名」が流れます。

- 終了: コンソールで `q` + Enter（または Ctrl+C）→ 集計を表示して
  `results-*.jsonl`（全クリックの記録）と `summary-*.json`（アプリ別集計）を保存します。
- **計測ツールのコンソール自体はクリックしない**でください（記録に混ざります）。

### 対象アプリ（4カテゴリ・計10種目安）

ロードマップで確定した4カテゴリすべてを回してください。1アプリ 2〜3分・20クリック程度で十分です。

| カテゴリ | 例 |
| --- | --- |
| ① Office系 | Excel / Word / Outlook |
| ② ブラウザ内の業務システム | Chrome / Edge 上でふだん使う Web アプリ |
| ③ 独自の業務デスクトップアプリ | 社内の業務アプリ |
| ④ Windows標準 | エクスプローラー / 設定 / メモ帳 |

## 結果の返し方

`results-*.jsonl` と `summary-*.json` の2つを開発セッションへ共有してください。

> ⚠️ **共有前の確認**: 記録には「クリックした要素の名前」「ウィンドウタイトル」が
> そのまま入ります（例: 開いていたファイル名・メール件名）。機密が含まれていないか
> 一読し、まずい行は削除してから共有してください（行単位で消して問題ありません）。

## うまく動かないとき

- **`npm run selftest` がエラーになる / measure.js が起動しない**
  → 予備の PowerShell 版で計測してください（Node.js 不要・記録形式は同じ）:

  ```bat
  powershell -ExecutionPolicy Bypass -File measure.ps1
  ```

  ※ selftest のエラーメッセージはそのまま開発セッションへ報告してください。
  koffi 経由の UIA 呼び出しが成立するか自体が、このスパイクの検証項目の1つです。
- **ブラウザ内の要素名が全滅する**
  → ブラウザのアクセシビリティが無効の可能性があります。UIA クライアントの接続で
  自動有効化されるのが通常ですが、変わらない場合は Chrome/Edge を
  `--force-renderer-accessibility` フラグ付きで起動して再計測してください
  （この場合、その旨をメモに残してください。R2 の設計判断材料になります）。
- **集計だけやり直したい**: `node summarize.js`（フォルダ内の results-*.jsonl を再集計）

## 2-R2c ダンプ手順（フォルダ名を正しく読むための追加計測）

エクスプローラーでフォルダをクリックすると要素名が列見出しの **「名前」** になってしまう
問題を、UIA ベースで直すための実機ダンプです（仕様は `docs/spec-2-R2c-folder-name.md`）。
`measure` とは別に、**クリック先の最深要素から親を上へたどった「親チェーン」**を記録します。
これで「フォルダ表示名を持つ `ListItem` がどの階層に居るか（＝方式A の登り停止先）」と
「アドレスバーの構造（＝方式B の読み取り経路）」、`TreeWalker` の vtable が実機で通るかを確定します。

```bat
npm run selftest:dump   … 動作確認（カーソル位置の要素＋親チェーンを1回表示）
npm run dump            … ダンプ計測開始
```

計測が始まったら、**次の①〜④を1回ずつ**クリックしてください（各1クリックで十分）。

1. エクスプローラーを **「詳細」表示**にして、**フォルダを1回クリック**（方式A の主材料）
2. **「中アイコン」または「大アイコン」表示**で、同じ**フォルダを1回クリック**
3. 左の**ナビゲーションツリーのフォルダを1回クリック**（`TreeItem` の確認）
4. どこかのフォルダを**ダブルクリックして入った**あと、**アドレスバーの「現在フォルダ名」
   部分を1回クリック**（方式B の主材料）

- コンソールには各クリックで「`最深: ...`」と、その上に「`↑1 ... ↑2 ...`」と親チェーンが
  流れます。`ListItem`「フォルダ名」の行が現れていれば方式A は成立見込みです。
- 終了: `q` + Enter（または Ctrl+C）→ `dump-*.jsonl` を保存します。
- `selftest:dump` で「親チェーンが空」と出た場合は、その旨（`get_ControlViewWalker` の
  結果）をそのまま報告してください。vtable インデックスの実機確定に使います。

### 返し方（2-R2c）

`dump-*.jsonl` を開発セッションへ共有してください。**フォルダ名・パス等の固有名が
含まれる**ため、共有前に一読し、まずい行は削除して構いません（行単位で消せます）。

## 判断基準（ロードマップより）

| 実測結果 | 判断 |
| --- | --- |
| Office・Windows標準で取得率 70%以上 | 2-R2 は計画どおり |
| ブラウザ内アプリが 50%未満 | UIA ブリッジ等を追加調査してから 2-R2 を設計 |
| 独自業務アプリが低い（想定内） | フォールバック文生成を主系統の一つとして設計 |
| 全カテゴリ平均3割未満 | 期待値を下げ、R3 は座標中心切り出し主体へ。到達レベル再協議 |
